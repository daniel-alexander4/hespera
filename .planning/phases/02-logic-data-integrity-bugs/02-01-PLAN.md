  <name>Task 1: Guard TV identity upsert against overwriting resolved/skipped rows</name>
  <files>internal/tvscan/scanner.go</files>
  <action>
In `tvscan/scanner.go`, modify the tv_series_identities upsert at lines 147-156. The current ON CONFLICT clause unconditionally overwrites guessed_title, season_number, episode_numbers_csv, match_confidence, and match_method. This destroys manually resolved or skipped identity data on rescan.

Fix the identified-file path (ident != nil, lines 145-159):
Change the ON CONFLICT clause to only update identity fields when the existing row status is NOT 'resolved' and NOT 'skipped'. Use this pattern:

```sql
INSERT INTO tv_series_identities (file_id, status, guessed_title, season_number, episode_numbers_csv, match_confidence, match_method)
VALUES (?, 'needs_fix', ?, ?, ?, ?, ?)
ON CONFLICT(file_id) DO UPDATE SET
  guessed_title=excluded.guessed_title,
  season_number=excluded.season_number,
  episode_numbers_csv=excluded.episode_numbers_csv,
  match_confidence=excluded.match_confidence,
  match_method=excluded.match_method
WHERE status NOT IN ('resolved', 'skipped')
```

The WHERE clause on the DO UPDATE means the update only fires when the existing row's status is neither 'resolved' nor 'skipped'. For new rows (no conflict), INSERT proceeds normally.

The unidentified-file fallback path (ident == nil, lines 160-168) already uses ON CONFLICT DO NOTHING, which is correct -- leave it unchanged.

Do NOT change the tv_series_files upsert (lines 117-126) -- file metadata (container, size, mtime, stream_info) should always be updated on rescan.

<task type="auto">
  <name>Task 2: Replace detached goroutine with job queue enqueue in tvMatchApprove</name>
  <files>internal/web/handlers_tv.go</files>
  <action>
In `handlers_tv.go`, modify the `tvMatchApprove` handler (lines 607-614). Currently it spawns a detached goroutine:

```go
matcher := tmdb.NewMatcher(h.db, h.cfg.TMDBAPIKey, h.cfg.DataDir)
go func() {
    ctx := context.Background()
    if err := matcher.FetchShowMetadata(ctx, tmdbID); err != nil {
        slog.Warn("fetch show metadata failed", "tmdb_id", tmdbID, "err", err)
    }
}()
```

Replace with a job queue enqueue. The challenge: `Enqueue` requires a `libraryID` but tvMatchApprove doesn't have one. The metadata fetch (`FetchShowMetadata`) doesn't need a libraryID -- it's a show-level operation. Use `libraryID=0` as a sentinel for "not library-scoped" jobs (the jobs system accepts it and the executor closure ignores it).

Replace the goroutine block (lines 607-614) with:

```go
// Fetch metadata via job queue (not detached goroutine).
matcher := tmdb.NewMatcher(h.db, h.cfg.TMDBAPIKey, h.cfg.DataDir)
capturedTmdbID := tmdbID
_, enqErr := h.jobs.Enqueue("tv_metadata_fetch", 0, "user", func(ctx context.Context, jobID, libraryID int64) error {
    return matcher.FetchShowMetadata(ctx, capturedTmdbID)
})
if enqErr != nil {
    slog.Warn("failed to enqueue tv metadata fetch", "tmdb_id", tmdbID, "err", enqErr)
}
```

Key points:
- Use job type `"tv_metadata_fetch"` to distinguish from `"tv_match"` (full match pipeline)
- Use `libraryID=0` since this is not library-scoped
- Capture `tmdbID` into `capturedTmdbID` to avoid closure variable issues (tmdbID is an int so it's already copied, but be explicit for clarity)
- The executor ignores `jobID` and `libraryID` params -- it only needs the tmdbID
- Log a warning if enqueue fails but don't fail the HTTP request (metadata fetch is best-effort, the identity resolution already committed)
- Remove the `context` import usage for `context.Background()` if it becomes unused (check other usages in the file first -- it's likely used elsewhere, so leave it)
