<objective>
Sanitize all HTTP error responses to eliminate internal detail leakage, and ensure every error is logged server-side.

Purpose: 109+ handler error paths currently pass raw err.Error() to HTTP clients, exposing SQL errors, filesystem paths, and Go internals. This plan introduces httpError/jsonErr helpers that log internally and return generic messages, then mechanically replaces every leaking instance.

Output: All handler files sanitized. Zero instances of err.Error() passed to http.Error or jsonError. All errors logged via slog.
</objective>

    Add two new helper functions to internal/web/handlers_core.go, placed after the existing jsonError function (line 125):

    1. `httpError` -- logs real error via slog, sends generic message to client:
       ```go
       func httpError(w http.ResponseWriter, code int, msg string, logMsg string, attrs ...any) {
           if code >= 500 {
               slog.Error(logMsg, attrs...)
           } else {
               slog.Warn(logMsg, attrs...)
           }
           http.Error(w, msg, code)
       }
       ```

    2. `jsonErr` -- same pattern for JSON responses:
       ```go
       func jsonErr(w http.ResponseWriter, code int, msg string, logMsg string, attrs ...any) {
           if code >= 500 {
               slog.Error(logMsg, attrs...)
           } else {
               slog.Warn(logMsg, attrs...)
           }
           jsonError(w, msg, code)
       }
       ```

    Note the signature: `(w, code, msg, logMsg, attrs...)`. The code comes SECOND (not third like in the research doc) so it reads naturally: "error to w, status 500, saying X, because Y".
