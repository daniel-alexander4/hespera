# CLAUDE.md

## Project Overview

Locally-hosted media app in Go: Music, TV, Movies with automatic metadata matching. SQLite storage, server-rendered HTML templates with vanilla CSS/JS. **Distributed as self-contained cross-platform binaries** (Linux/macOS/Windows × amd64/arm64) — the `web/` UI assets are embedded via `//go:embed` (`embed.go`, `hespera.WebFS()`), so a binary finds its templates/static regardless of the working directory; a Docker image is still provided. **Runs as a single-machine app by default**: launching the binary opens a chromeless app-mode browser window (`internal/browser`, nib-ported) bound to a **random loopback port** (a single-machine app has no access-control layer — loopback needs none). `HESPERA_NO_BROWSER=1` is server mode (Docker/headless — no window, honors `HESPERA_LISTEN`). The desktop `.deb` ships a `.desktop` entry + icon (`Exec=hespera --replace`; `internal/singleton` `/proc`-scans to replace a running instance so a relaunch doesn't collide on the port). The topbar carries **reload** + **full-screen** + **power** buttons: the full-screen button toggles the real Fullscreen API on the app root (`documentElement`) — and while app-full-screen the topbar becomes an **auto-hiding overlay** (reveal on mouse/keyboard, hide after ~2.5s, mirroring the media-player controls; the JS is a window-guarded IIFE in `layout.html`, delegated so it survives Turbo swaps). Gated on `html.is-fullscreen`, which is set **only** when `document.fullscreenElement === documentElement` so the per-video full-screen (kept as-is) never triggers the topbar overlay; on a TV this overlay IS the topbar behavior (full-screen hides it, remote keys reveal it — there is no couch mode). On a page with its own play controls (the music now-playing `#player-transport`, or a TV/movie `[data-media-kind]` video → the JS sets `html.has-play-controls` on `turbo:load`), the full-screen topbar overlay is **suppressed entirely** (the play controls are the only chrome); windowed player pages keep the topbar. power → `POST /shutdown` (same-origin-guarded; `web.Deps.Quit` → main's `quit` channel → same graceful path as SIGTERM) quits the app. `browser.Open` launches the app window in a **dedicated profile** (`DataDir/browser`, so the launched process owns its window) and main SIGTERMs it on quit to close the window; the client also calls `window.close()`. Four direct deps: `dhowden/tag`, `modernc.org/sqlite`, `bogem/id3v2/v2`, `gcottom/audiometa/v3`. Licensed **GPL-3.0** (`LICENSE`); third-party attributions in `THIRD_PARTY_LICENSES.md`, kept current by `TestThirdPartyLicensesCurrent` (`cmd/hespera`) — it fails the build if a direct dep in `go.mod` isn't listed there, so **adding a dependency requires updating `THIRD_PARTY_LICENSES.md`**.

## Build & Run Commands

```bash
# Build binaries locally (quick dev build → ./bin)
go build -o ./bin/hespera ./cmd/hespera
go build -o ./bin/hescli ./cmd/hescli
make build                  # same, via the Makefile

# Cross-compile + package for release (mirrors ~/repos/nib: bash + nfpm, no goreleaser)
./build.sh                  # 6 binaries (Linux/macOS/Windows × amd64/arm64) + 2 .deb → dist/
./install.sh                # `apt install` the prebuilt dist/ .deb on this Debian/Ubuntu box (run build.sh first)
make dist / make install    # Makefile wrappers
# build.sh stamps the version via -ldflags "-X main.version=$VERSION" from the VERSION file;
# the .deb (build/nfpm.yaml) ships hespera+hescli to /usr/bin and declares ffmpeg.
# There is no service: the binary runs on demand as the invoking user (see the daemon note below).

# Build and run with Docker (still supported; bundles ffmpeg)
docker compose up --build

# Tests: all / one package / one test
go test ./...
go test ./internal/config
go test ./internal/config -run TestFromEnvDefaults

# Format and vet
go fmt ./...
go vet ./...

# Bump the semantic version (X.Y.Z in the VERSION file)
./bump.sh patch            # Z+1 — a minor change / fix
./bump.sh minor            # Y+1, Z=0 — a major feature
./bump.sh major            # X+1, Y=Z=0 — a breaking release
make bump-patch / bump-minor / bump-major   # Makefile wrappers
```

> **Versioning (SemVer `X.Y.Z`).** The `VERSION` file is the single source of
> truth; `build.sh`/`install.sh` stamp it into the binary via
> `-ldflags "-X main.version=…"` (it also drives the `staticv` asset
> cache-buster). **Bump policy:** **patch (Z)** for a minor change or fix,
> **minor (Y)** for a major feature, **major (X)** only when Dan says. Bump as
> part of the change's commit (use `./bump.sh <part>`); Y/Z are chosen by the
> change's size, X is never bumped automatically.

> **Binary vs Docker distribution.** The standalone binary runs as the invoking
> user (no container, no root, **no systemd service**) and, by default, **opens an
> app window** (`browser.Open`) on a random loopback port — a single-machine app,
> not a LAN server. `HESPERA_NO_BROWSER=1` opts into server mode (Docker/headless).
> Three consequences baked into the code: (1) `config` defaults `DataDir` to a
> per-user dir (`os.UserConfigDir()/hespera`) and validates paths with
> `filepath.IsAbs` (not a `/`-prefix), so Windows works; `main` `MkdirAll`s the
> data dir on first run. (2) `web.New` loads templates via `template.ParseFS` and serves `/static`
> + favicon from the embedded `hespera.WebFS()`; the static cache-buster
> (`staticv`) is the build version, not a file mtime. Tests inject a stub asset
> tree via `web.Deps.AssetsFS` (nil → embedded). (3) **`web.New` resolves one
> settings-overridable value once at construction via `resolveEffectiveConfig`**
> (`effective_config.go`): the **media folder** (`app_settings.media_root`). It
> overrides the handler's `cfg` copy before any scanner is
> built — and since every scanner is `scan.New(h.cfg, …)` and both stream handlers
> read `h.cfg.MediaRoot`, this **one override is the single source of truth** (no
> call site reads `app_settings` for it; the pathguard containment code is
> untouched). It applies at the next launch (the Settings form says "applies on
> restart"), keeping the security boundary resolved at construction, not per
> request. A saved-but-invalid media folder warns and falls back to the
> env/default rather than failing to boot. Docker is unaffected — its compose sets
> `HESPERA_NO_BROWSER`/`HESPERA_LISTEN`/`HESPERA_DATA_DIR`/`MEDIA_ROOT`
> explicitly.

## Architecture

> **Before any work that touches media data** (scanning, matching, pruning,
> playback state, artwork, or any query/handler that reads or writes a media
> row), consult [`docs/media-source-of-truth.md`](docs/media-source-of-truth.md)
> — the canonical reference for what owns each piece of data per media type,
> how files map to DB rows, and what survives vs. is lost across
> rescan/rematch/move/delete. Keep it in sync when that behavior changes.

### Entry Points

- `cmd/hespera/main.go` — Web server: config → SQLite (WAL) → migrations → Handler → HTTP server, graceful shutdown (10s timeout). Also starts the **hescli management socket** (Linux only, best-effort): a unix socket at `DataDir/hescli.sock` serving `web.Handler.ManagementRouter()`, gated by a `peerCredListener` (`SO_PEERCRED` via stdlib `syscall.GetsockoptUcred` — **no new dep**) that admits only **root or the server's own uid** (`authorizedUID`); the socket is 0600, stale-removed on start, unlinked on shutdown. Build-tagged (`management_linux.go` + a no-op `management_other.go`) since peer-cred is Linux-only. The socket path is derived by **`config.ManagementSocketPath(dataDir)`** (the single definition of the `hescli.sock` name): `DataDir/hescli.sock` when it fits the ~108-byte unix `sun_path` limit, else a deterministic `$XDG_RUNTIME_DIR` (or temp-dir) fallback named `hespera-<hash-of-DataDir>.sock` — server and hescli both call it, so they agree on the fallback with no coordination and distinct DataDirs never collide.
- `cmd/hescli/main.go` — **management CLI** (headless configuration over SSH). An HTTP client over the management unix socket (`--socket`/`$HESPERA_SOCKET`/default `DataDir/hescli.sock`, resolved via `config.FromEnv()`); always sends `Accept: application/json`. Verbs: `library list|add|rm`, `scan`, `match`, `integrity`, `config list|get|set`, `jobs`, `status` — **no playback**. Reuses the web handlers that already JSON-branch on `requestWantsJSON` (scan/delete/integrity/jobs); the rest are thin JSON handlers in `internal/web/management.go`. `config set` drives a small `managedSettings` registry (secret/string/toggle/path kinds) over `saveAPIKey`/`validateMediaFolder` — `media_root` prints "applies on restart".

Actor profile images stay in the shared `thumbs/tv` (`Matcher.personDir`, distinct from `artDir`) since people are global |
| `internal/tvscan` | TV file identification — an ordered most-specific-first cascade (`identify.go`): SxE with separators + multi-ep ranges (`S01E01-E02`) → N×M with ranges (guarded by leading `\b` + 2-digit episode so `1280x720`/`4x4` never parse) → year-first air-date → season-dir-relative markers (`Episode N`/E-only/`N of M`/`- 01 -`/bare-number, **only under a season dir**) → folder-authoritative season-dir fallback. `ParseSeasonDir` accepts `Season N`/`sN`/`Series N`/`Saison`/`Staffel`/`Temporada`/`Specials`→0 (not bare numbers or `Extras`); `cleanTitle` strips leading `[group]`/`{id}`/`www.site -`. **Folder-year preference** (`resolveTitleYear`): a year carried by the *show folder* (`Doctor Who (2023)` — the season-dir's parent, else the file's dir) is stripped from the title and recorded as `EpisodeIdentity.Year` (→ `tv_series_identities.year`); the matcher groups by `(title, year)` and `pickBestResult` adds a first-air-year bonus, so reboots that share a name (Doctor Who 1963/2005/2023) match the right era. Era-titled shows like `1883` keep their title (a leading bare year isn't taken); no-year folders are unchanged. Scanner-level junk-skip (`IsJunkFile`/`IsJunkDirName`): `.sample.` files + nested `Sample`/`Extras`/`Trailers`/`Featurettes` dirs (top-level same-named shows survive). Scanner writes `stream_info_json` = marshaled `video.ProbeResult`. A separate `tv_probe` job (`ReprobeMissing` in `reprobe.go`, **auto-chained after every TV library scan**, last in the scan→match→integrity chain; the former libraries-page "Verify playback" button was removed) re-probes files whose `stream_info_json` is empty (`''`/`{}` — a scan-time probe failure, often a transient ffmpeg-semaphore acquire timeout, or rows predating the column; the size+mtime fast-path won't re-probe an unchanged file) so the seekable HLS path always has the source duration it needs up front. **Per-series scan**: `ScanTV` is built on a shared `scanTVRoots(roots)` engine (+ extracted `ingestTVFile`); `ScanTVDirs(dirs)` walks only a series' show folder(s) with relink/prune scoped **per dir** (so only rows under the scanned folder can be pruned), and `ShowDirsForFiles` maps a series' file paths → its show folder(s) (the season-dir's parent, else the file's dir — scanning the *show* folder is what finds a brand-new `Season N/`). Powers the series-page **Scan for new episodes** button (`tvSeriesScan`, `POST /tv/series/scan` → scoped scan → chained `tv_match`; full-scan fallback if folders are absurdly scattered) |
| `internal/moviescan` | Movie file identification + scan — a strict simplification of `tvscan` (`ScanMovies` mirrors `ScanTV`: walk/probe/junk-skip via reused `tvscan.IsJunkFile`/`IsJunkDirName`, pathguard, move-relink on `(size,mtime)`, prune), minus the SxE cascade. One new piece: a **Title+Year parser** (`ParseMovie`, `identify.go`) — parenthesized year preferred, else last in-range bare year; folder-authoritative for `Movie (2019)/clip.mkv`; tolerates titles that contain a year (Blade Runner 2049). Single-table upsert into `movie_files`; the match columns are matcher-owned and never written here. A `movie_probe` job (`ReprobeMissing` in `reprobe.go`, the movie twin of `tvscan`'s, auto-chained after every movie scan) backfills empty `stream_info_json` |
| `internal/video` | ffprobe wrapper + gated ffmpeg execution (`StreamFFmpeg`, `EnsureSegment`, `VODPlaylist`), concurrency caps, HLS cache. Also `ProbeTags`/`ExtractCoverArt` (gated) — the audio reject-fallback used by `internal/scan` to recover tags + embedded cover when the pure-Go reader rejects a non-MP3 file. Also `Fingerprint`/`ChromaprintAvailable` — Chromaprint audio fingerprints via ffmpeg's built-in `chromaprint` muxer (no extra binary), feeding intro detection. Also `CheckIntegrity`/`RemuxCopy`/`RepairFile` — the media corruption detection + lossless container-repair primitives driven by `internal/integrity` (gated) |
| `internal/integrity` | Media corruption detection + lossless auto-repair for **video (TV + movies) and music**. Per-library orchestration around `video.CheckIntegrity`/`RepairFile`; **the one place Hespera writes back into `MEDIA_ROOT`** (everything else treats media files as read-only). **Two tiers:** (1) `CheckLibrary` (cheap, chained after a scan) demuxes each new/changed file (`-c copy -f null`, no decode) to find **container** corruption and — gated by the `integrity_autorepair` kill-switch (default on) — losslessly remuxes it (`RemuxCopy`) to a same-dir hidden temp, **verifies** (`remuxVerified`: same stream count + duration within ±2s + a clean re-check), then **atomically renames over the original** and refreshes the row's size/mtime/`stream_info_json`; a verify failure discards the temp and leaves the original untouched. It **also runs the cheap audio-gap scan** (shared `auditAudio`→`video.AudioGaps`) on every file, so new media is examined for **both** container corruption (repaired) **and missing audio** (flagged) automatically on scan — the video full-decode is the only thing left to the opt-in deep tier for cost. (2) `DeepCheckLibrary` (opt-in "Check integrity" button) additionally full-**decodes** for **video bitstream** corruption (damaged frames a demux can't see) alongside the same audio-gap scan (`video.AudioGaps`, demux-only, ≥`audioGapFlagThreshold`=0.5s) — **flag-only**, since that data is unrecoverable (the transcode gap-fill silence-pads it for playback; the file is still damaged). Flag detail distinguishes audio vs video. Per-file status on `{tv_series_files,movie_files,music_tracks}.integrity_status` (`''`/`ok`/`repaired`/`flagged`), reset to `''` by the scanner on a size/mtime change. **Music (`music_tracks`) differences:** (a) the post-repair refresh is **table-aware** (`refreshAfterReplace`) — video writes `stream_info_json`, music recomputes `checksum_sha256` (the move-relink signature, which the remux changed) via `sha256File`; (b) music files are small, so the **full decode runs in the cheap scan-time tier for music** (`repairOne` calls `auditDecode` when `table=='music_tracks'`) — MP3 corruption surfaces only on decode, not in the container, so a container-only check would miss it; video keeps the full decode in the opt-in deep tier (minutes/file). Container repair still applies to music (helps FLAC/M4A/OGG; MP3 has no container to repair → its corruption is flag-only bitstream). Best-effort, gated by the shared ffmpeg semaphore. Repair logic is seam-injected → unit-testable without ffmpeg; a gated `TestRepairFileLive` (`HESPERA_LIVE_FIXTURE`) exercises real ffmpeg on a copy |
| `internal/introskip` | Marker-less intro detection: cross-matches a TV season's per-episode audio fingerprints (bounded brute-force shift + per-point Hamming run-length, `DetectIntros`) to find the shared title theme at each episode's own offset. Pure data — the fingerprint extractor is injected (no `video` dep), so it's fully unit-testable. Powers the "Detect intros" button (`tv_skip_segments` → skip-intro) |
| `internal/thumbgc` | Orphaned-thumbnail sweep (`Sweep`): deletes images under `thumbs/{music,tv}` referenced by no `art_path` column, with a grace window. Run as the final step of each match job (single-worker queue serializes it against art writers); reference set = `music_albums`+`music_artists` for music, `tv_series_art`+`people` (actor profile images) for TV |
| `internal/playback` | Pure TV playback-decision layer: per-client container↔codec matrix → direct-play / remux / transcode |
| `internal/opensubtitles` | OpenSubtitles REST v1 client (`OSClient`): key-only auth, search by TMDB series id + season/episode, download link. Inert (nil) without a key. Powers on-demand TV subtitle search |
| `internal/web` | HTTP handlers, routing (`http.ServeMux`), template rendering, logging middleware, TV streaming endpoints |

### Key Patterns

- **Handler DI**: `web.Handler` receives `web.Deps{Cfg, DB}`; `web.New(d)` compiles templates and starts the job service.
- **Routing**: stdlib `http.ServeMux`, routes registered in `web.Router()`. The whole mux is wrapped **unconditionally** by `csrfGuard` (`csrf.go`) — a same-origin check (`isSameOriginUnsafeRequest`: Origin/Referer vs Host) that rejects cross-origin unsafe (POST/PUT/…) requests with a 403. There is **no authentication layer** — Hespera is a single-machine loopback app (a LAN-exposed server has no access control; that's a deliberate posture, not an oversight).
- **Templates**: `html/template` from `web/templates/` at startup. Layout base cloned per page, merged with `partials_*.html` glob. FuncMap: `staticv` (cache-bust), `humanBytes`, `mult`.
- **Icons**: inline Lucide SVGs defined in `partials_icons.html` (auto-loaded by the `partials_*.html` glob); use as `{{template "icon-NAME"}}`. Monochrome stroke glyphs inheriting `currentColor`, so they theme with Catppuccin automatically; sized via the `.icon` CSS class in `app.css`. Applied to the topbar nav, the stable music pages (album/albums/artist/compilations action buttons + `♪` art placeholders), and the `player.html` transport controls; TV/Movies/Settings/`music_home` are not yet converted.
- **Persistent music player (Turbo Drive)**: vendored Hotwire **Turbo Drive** (`web/static/turbo.umd.js`, MIT) turns full navigations into `<body>` swaps, so the single `<audio id="hespera-audio" data-turbo-permanent>` in the `layout.html` shell **keeps playing across page changes**. `web/static/player.js` is the one global controller: it owns that audio + the queue/shuffle/index state, the topbar **now-playing cluster** (`#np-cluster`, play/pause + track title linking to `/music/player` + a stop **X** (`#np-close` → `stopPlayback`: stops playback, clears the queue, hides the cluster), `data-turbo-permanent`, hidden when idle), Media Session, `/music/play-event` reporting, and — when the now-playing view (`player.html`, a static `.player-page` shell) is on the page — its cover/seek/transport/playlist/synced-lyrics. Play actions are `[data-play]` controls (album/artist/home + the era `[data-play-form]`), each an `<a href="/music/player?…">`; **initiating playback navigates to `/music/player`** (Turbo), whose `.player-page` shell **autoloads** the queue from the URL params (`data-autoload` → `GET /music/queue`, JSON; same `?source=`/`album`/`era`/`popular` params built by the one `buildPlayerQueue`) and starts it — so starting a song *takes you to the now-playing page* (the Turbo-permanent audio + the `#np-cluster` breadcrumb handle wandering off and coming back). player.js only plays **in place** (intercepts the click) when you're already on `/music/player`. Autoload fires on **any** params-bearing visit, so starting a new song while one plays swaps the queue; the bare `/music/player` breadcrumb (no params) doesn't reload. `loadQueue` supports `startTrackId` and `startTitle`+`startArtist`+`keep` to start at a specific song — a queue can start at the **clicked** song, keeping earlier ones queued as "previous" (matching by title+artist). Turbo lifecycle adaptations: `couch.js`/`subtabs.js` re-init on `turbo:load` (couch keydown still added once); the `layout.html` dropdown-menu script binds its document click/keydown listeners **once** (window-guarded) because the inline script re-runs on every render — otherwise they'd accumulate one pair per swap; pages with side effects tear down on `turbo:before-cache` (`once:true`) — `tv_player.html` (hls.js worker; also `video.pause()` + clears `src` so a detached element can't keep playing), `settings_jobs.html` (poll), `tv_match_review.html` (document listener); `hls.light.min.js` moved to the shell so it isn't re-evaluated per swap. Starting any other media (a TV video) pauses the music; conversely, **leaving** a TV **or movie** player surfaces a topbar **resume chip** (`#nw-cluster`, `web/static/tv_resume.js`, `data-turbo-permanent`, sessionStorage-backed) linking back to `/<kind>/player?file=N` — it records `{kind,fileID,label}` (the TV and movie file-id namespaces differ, so the kind disambiguates them; a stale keyless entry defaults to `tv`); resume position is stored server-side per kind, so the link only needs the file id. All POST handlers already 303-redirect, which Turbo requires.
- **Landing dashboard** (`home`, `home.html`): the `/` route is a cross-library dashboard, not a static launcher. Keeps the three nav cards (Music/TV/Movies) at the top (Libraries lives in the Settings menu), then renders activity rows that **reuse existing queries + the `band-album-card` carousel** — Continue Watching (`recentTVSeries(tvRecentlyWatchedQuery)`, deep-links to the in-progress season), Recently Played (`loadRecentlyPlayedArtists`, factored out of `musicHome`), Recently Added Albums (`loadRecentlyAddedAlbums`), Recently Added Shows (`recentTVSeries(tvRecentlyAddedQuery)`) — plus a Quick Play strip (`[data-play]` Shuffle All / Most Popular) and a stats line (`loadHomeStats`). Every section is best-effort (a failed query warns and renders empty, never 500s the page). Movies now appears in the content rows too — a **Recently Added Movies** row (`loadMovieRecentlyAdded`) + a movies count in `loadHomeStats`.
- **Music quick-play (music-home Recent tab, `music_home.html`)**: Shuffle All + Shuffle Most Popular (`[data-play]` links) + a **Shuffle Era** range picker — all over the local library (`source=all`/`source=popular`/`source=era` in `buildPlayerQueue`). The era picker (`web/static/era_slider.js` + `.era-*` CSS) is a **transparent lens that slides + resizes over a measuring-tape of years** (a per-year tick, taller every 5, tallest + a full-year number (1970, 1980, …) every 10, drawn behind the glass so the tape reads through it) — replacing the old from/to `<select>`s (which were slow on a remote — couch.js steals arrows from a focused `<select>`): drag the body to slide the range, drag an edge to resize it; **remote/keyboard** — with the `.era-track` focused, ◀▶ slide and ▲▼ grow/shrink the span from centre, Enter plays (no on-screen hint line). Its keydown `stopPropagation`s so couch's document-level focus-mover never sees the arrows. Always-on decade tick labels keep the range targetable. Two buttons ride beside the slider — **Play** (era in order) and **Shuffle** (adds `&shuffle=1`; shuffle is applied client-side in `player.js`, so both hit the same ordered `source=era` query). Pure client-side: it keeps both `[data-play]` hrefs in sync with the range; the backend era query is unchanged. The picker is a **shared partial** (`web/templates/partials_era_picker.html`, `{{template "era-picker" .EraPicker}}`) rendered on the music-home Playlists tab **and the Home Quick-Play card**, fed by the one `h.eraPicker` helper (a `MIN/MAX(year)` query → `eraPickerData`, nil when no year-tagged albums). _The dedicated `/music/playlists` hub page (`musicPlaylists`, `music_playlists.html`) and the Billboard/YouTube **Top 100** feature were both **removed** — the hub was redundant once Top 100 was gone (its shuffle controls already live here). Top 100 was structurally unplayable: YouTube blocks embedded music playback in signed-out browsers (onError 150), and Hespera's app window is always a dedicated signed-out Chrome profile. `internal/{billboard,youtube,ytdlp}`, `popout.html`, `playlists.js`, and the `youtube_lookups`/`itunes_art` caches went with it._

### Match Pipeline

- **MBClient**: MusicBrainz API, 1 req/sec rate limiter, 3-strategy query cascade (strict release-group → loose release → artist fallback). Strategy A fetches 25 candidates so the canonical studio release-group isn't crowded out by compilations/EPs of the same title.
- **Scorer**: weighted (title 0-38, artist 0-26, MB score 0-18, type 0-10, year 0-4; max ~96). Single threshold `matchThreshold` (=80). The former "uncertain" tier was retired (migrated to unmatched in `db.Migrate`). `typeBonus` penalizes EP/Single and any non-primary edition — including **secondary types** (Live/Compilation/Remix/Demo on a primary=Album RG) — so a clean studio album outranks art-less alt editions of the same title and the matcher picks the release-group that actually has Cover Art Archive art. The penalty is **non-demoting** (`BestMatchCandidate`): a candidate is eligible when `ScoreCandidate + typeDemotion ≥ matchThreshold` (its score crediting a *clean* type), so the edition penalty only reorders same-titled siblings and never unmatches a strong title/artist/year match (a real Live album like *Made in Japan* stays matched); among eligibles the highest actual score wins, keeping the studio-album-for-art preference. Strict superset of the old `ScoreCandidate ≥ 80` gate — common path unchanged, only near-threshold secondary editions are rescued.
- **CAAClient**: Cover Art Archive, release-group → release fallback, thumbnail size preference. Multi-candidate art search: if the matched release-group has no front cover, `fetchAlbumArt` (`pipeline.go`) probes sibling above-threshold candidates — gated to **same-artist, clean primary=Album, within 8 score points** (so only a same-album edition's cover is reused, never a live/compilation/different-album), RG-level only, capped at 3.
- **Manual art override**: `POST /music/album/art` (form on the album edit page) uploads a cover image when CAA has none or the album mis-matched. Validates server-derived MIME (jpeg/png/webp only), caps 15 MiB (`MaxBytesReader`), writes an album-id-keyed file under `thumbs/music` (temp+rename) and sets `art_path` unconditionally. The scanner/matcher art writers are empty-only-guarded, so manual art survives rescan/rematch. Served via `/art/album/{id}` with `X-Content-Type-Options: nosniff`. Upload-only (no fetch-by-URL — SSRF).
- **Artist enrichment**: MusicBrainz URL relations → Wikipedia REST API (bio) → Wikidata/Wikimedia Commons (image). Optional MBID-keyed backfill **only where those leave a gap**: fanart.tv (artist image, `fanart.go`) then TheAudioDB (bio + image, `audiodb.go`). Both opt-in via user-supplied keys (own per-host limiters, nil/no-op without a key), keyed by artist MBID so they stay correct even when album release-group MBIDs mis-match. Empty-only writes preserved. Auto-resolution (`SearchArtist`) blindly takes the top name-search hit, which picks the wrong artist when several share a name (e.g. the 1897 country "Jimmie Rodgers" over the 1933 pop singer) — the **manual artist disambiguation control** (below) is the user-facing fix.
- **Manual artist disambiguation** (`handlers_music_artist.go`, "Fix artist match" on the artist page): `GET /music/artist/disambiguate?id=N` renders MB `SearchArtistCandidates` (up to 10, with disambiguation/type/country/life-span/score, current one flagged); `POST` validates the chosen MBID against a UUID pattern, sets `music_artists.musicbrainz_id`, clears bio/bio_source/art_path, then **re-enriches synchronously** (`Matcher.ReEnrichArtist` → `EnrichArtist` for the chosen MBID) so the corrected bio/image show immediately (non-fatal on network error — next Match fills the gap). Touches only `music_artists`; per-album `artist_musicbrainz_id` (resolved per release-group, individually correct) is left untouched.
- **Manual artist image picker** (`handlers_music.go`, "Artist image" on the artist page): `GET /music/artist/art?id=N` renders selectable image candidates — fanart.tv's full gallery (all thumbs+backgrounds, `FanartClient.ArtistImages`) + TheAudioDB thumb, aggregated by `Matcher.ArtistImageCandidates(mbid)` (providers nil/no-op without a key) — plus the current image and a file-upload form. `POST` applies a pick (re-fetches the candidate set and requires the chosen URL be a member — the **SSRF guard**; the server never fetches an arbitrary client URL), an uploaded file (MIME-from-bytes, jpeg/png/webp), or a clear. Writes `music_artists.art_path` **unconditionally** (temp+rename, keyed `manual-artist-<id>`) so the choice survives re-match; decoupled from MBID/bio. Served via `/art/artist/{id}` (now `X-Content-Type-Options: nosniff`). Without a fanart.tv key the gallery is just TheAudioDB + upload.
- **Track popularity (ListenBrainz)**: `fetchPopularity` (a phase of `RunMusicMatch`, after `enrichArtists`) fills `music_tracks.popularity` = global listen count from ListenBrainz's `/popularity/top-recordings-for-artist/{mbid}` (`listenbrainz.go`, `LBClient`, no key, shares the MetaBrainz 1 req/s limiter). Per artist-with-MBID it fetches top recordings and credits each local track whose `NormalizeForDedup` title **exactly** matches a recording name with that recording's count; unmatched tracks stay `popularity=0`. Best-effort/non-fatal, re-runs each Match (no TTL yet — fine at this library size). Powers the **Shuffle Most Popular** playlist (`source=popular`: per-artist top-`popularPerArtistLimit`=10 by popularity, pooled + shuffled, excludes `popularity=0` — **except** an artist with ≤`popularIncludeAllMaxTracks`=4 library tracks, a curated artist whose *whole* catalog is pooled regardless of popularity). **Optional Last.fm blend** (`internal/match/lastfm.go`, `LastfmClient`, key-gated via `lastfm_api_key`): when a key is set, `fetchPopularity` runs a second pass after ListenBrainz — `artist.getTopTracks` (read-only, API key only, no shared secret) fills `popularity` for tracks ListenBrainz left at **0** (LB stays primary; an LB-credited or already-counted track is left alone). Threaded through `match.New(db, dataDir, fanartKey, audiodbKey, lastfmKey)`.
- **Similar artists + out-of-catalog artist pages**: the artist page (`musicArtistAlbums`) shows a **"Similar Artists"** section — cards for related artists from the ListenBrainz Labs **similar-artists** model (`LBClient.SimilarArtists`, keyed by the artist MBID, the `algorithm` param pinned as a constant; the labs host has its own `labsURL` seam). The list is cached on `music_artists.similar_json` and lazily backfilled on first view via a keyless `artist_similar_fetch` job (`enqueueMusicFetch` — the music twin of `enqueueMetaFetch`, no TMDB key), gated by `similar_fetched_at` so it doesn't re-enqueue per render. Cards for similars **in** the catalog deep-link to the local artist page; the rest link to **`/music/artist/external?mbid=…`** (`musicArtistExternal`) — the music analogue of the TV actor page: a dedicated page for an out-of-catalog artist (bio + image + notable release-groups), cached in the global **`external_artists`** table (mirrors `people`) and lazily filled by an `external_artist_fetch` job (`Matcher.ResolveExternalArtist`). Its image is an **external hotlink** (`referrerpolicy=no-referrer`), not downloaded — so it can't be reaped by the thumbgc music sweep, which only references on-disk art. `EnrichArtist` was refactored to share `enrichArtist(…, download bool)`: in-catalog enrichment downloads to disk (`ImagePath`) as before; the external path takes `download=false` and keeps just the resolved `ImageURL`. Notable releases come from a new `MBClient.BrowseArtistReleaseGroups` (browse by artist MBID, primary-type Album). The route is registered before the `/music/artist/` prefix; the MBID is `mbidPattern`-validated; an MBID that turns out to be in-catalog 302s to the local page.
- **Pipeline**: `Matcher.RunMusicMatch()` = enrich artists → **fetch popularity** → match albums → **re-fetch missing art**. The matcher only processes `''`/`unmatched` albums, so a third pass (`refetchMissingArt`) re-runs *only* the cover-art step for `matched` albums still lacking `art_path` — anchored to the album's **stored** MusicBrainz identity (re-search supplies candidate breadth only), gated by an `art_checked_at` TTL (30d) so genuinely art-less albums aren't re-probed every run, and race-guarded on `musicbrainz_id`. This is why a rebuild's matching improvements show up on the next Match. Non-fatal per-album errors.
- **Manual album controls** (album edit page): `POST /music/album/art` (upload cover), `/music/album/art/clear` (clear `art_path` + `art_checked_at` → re-fetched next Match), `/music/album/unmatch` (full reset: identity + art → re-matched next Match), `/music/album/reassign` (manual release-group reassignment: paste a correct RG MBID → sets `music_albums.musicbrainz_id`, keeps it `matched`, clears art, then synchronously re-fetches the cover for the new RG via `Matcher.RefetchAlbumArt` — the album analogue of the artist-disambiguation control; the zero-FP fix for a wrong-RG match like terse "Grease" losing to an art-less stub, recovering art without a manual upload), `/music/album/rescan` (re-read the album's files from disk via `scan.ScanFiles` — synchronous, re-extracts embedded art under the empty-only guard; same path the tag-edit POST uses). All POST-only under `/music/` (same-origin CSRF); the reassign MBID is UUID-validated (`mbidPattern`).
- **Per-track tag editor** (`/music/track/edit?id=N`, the **Edit** button on each track row of the album page): GET pre-fills one track's Title/Artist/Album/Album Artist/Year/Track#/Disc# from the DB; POST writes them to the file via `music.WriteTrackTags` then `scan.ScanFiles` on just that file. Because `ensureAlbum` re-derives album membership from tags, editing Album/Album Artist/Year **reparents** the track to the resolved album and `cleanupEmptyAlbums` GCs an emptied phantom album (this is the UI fix for stray-tagged tracks like a single file mis-tagged into a 1-track phantom album). Empty/zero fields are left unchanged on the file (not blanked). A missing/moved file redirects back with a clear `error=1` (no silent no-op, unlike the album-edit whole-album path). Known `WriteTrackTags` gaps: Year is a no-op on OGG/Opus, MBIDs are MP3-only.

### TV Streaming

- **Decision** (`internal/playback`): pure, per-client. `Decide()` validates the container↔codec *combination* (not independent codec sets) and returns direct-play / direct-stream (remux) / transcode, failing safe toward transcode on any uncertainty. Text subtitles deliver as a WebVTT sidecar (extracted on demand at `/stream/tv-subtitles/`); all three convert sites (embedded extract for TV+movies, OpenSubtitles convert) run ffmpeg→VTT through one buffered helper `extractVTT` that **overlap-clamps** each cue's end to the next cue's start (`clampVTTOverlaps`, `subtitle_vtt.go`) so two cues can never be active at once — and buffering means a mid-stream ffmpeg failure can't half-write a 200. Bitmap subs (PGS/DVD/DVB) are **burned in**: `SubtitleBurnIn` forces transcode, the player offers them labeled "· burn-in", and the session routes them to a continuous progressive transcode (`/stream/tv-burnin/`, protocol `file`; see Execution). A text sidecar (incl. an OpenSubtitles fetch) is preferred when available — no transcode. Profiles for chrome/firefox/safari with UA inference.
- **Execution** (`internal/video`): `StreamFFmpeg` (gated, kill-on-disconnect) for remux/subtitles/bitmap-subtitle burn-in; **segment-on-demand HLS** for transcode. The manifest is a synthetic **VOD** playlist (`VODPlaylist`): every `hls_time` segment listed up front from the source's stored ffprobe duration, with `#EXT-X-ENDLIST`, so the player knows the full episode length immediately and can **seek anywhere** — no transcode needed to serve it. Each `segNNNNN.ts` is transcoded lazily on first request (`EnsureSegment` → `SegmentArgs`: accurate input `-ss`, forced keyframe on the first frame, `-output_ts_offset` to place it on the full timeline, `-fps_mode cfr` so each full segment is exactly `hls_time` of **video** → manifest EXTINF stays accurate with no cumulative drift, and **`-bf 0` (no B-frames)** so each segment's DTS==PTS and starts exactly on its `-output_ts_offset` boundary), so a seek costs only that one segment's encode regardless of position. **The `-bf 0` is load-bearing, not an optimization:** with B-frames the reorder makes a segment's first DTS land ~1 reorder-depth *before* its boundary, so adjacent segments overlap in DTS and Chrome's MediaSource rejects the append (`MediaError 3 — "Parsed buffers not in DTS sequence"`, playback never starts — worst on a mid-timeline resume, where the first appended segment fails outright). It only bit files that **must** transcode (ac3/eac3/HEVC) **and** have B-frames; direct-play/remux/burn-in are unaffected. **Per-segment CPU is capped two ways** so on-demand transcode doesn't peg the box: each encode is **`-threads 3`** (one segment ≈ 3 cores, not all of them), and a dedicated **segment-transcode semaphore** (`SetSegmentConcurrency`, default **1**, env `HESPERA_HLS_SEGMENT_CONCURRENCY`) serialises segments — hls.js prefetches a burst to fill its forward buffer, and without the cap each burst spawns a transcode per segment and they **collectively saturate the CPU** (the spiky graph). Serialising keeps the peak to a single encode while still filling the buffer faster than realtime (the buffer stays 30-45s ahead). The segment semaphore is a sub-limit *beneath* the general `FFmpegConcurrentLimit` gate (acquired outer-then-inner, consistent order → no deadlock) and, unlike the general gate, **waits without the 2s fail-fast** — a queued prefetch segment legitimately waits its turn (bounded by `segBuildTimeout`), where a fast-fail would 500 the later segments in a burst. (`-threads` changes encoded bytes → bump `segEncodeVersion`; the concurrency cap is pure scheduling → no bump.) The segment cache key (`hlsKey`) folds in a **`segEncodeVersion`** constant — bump it whenever `SegmentArgs` changes the encoded bytes incompatibly, so stale segments from an older encoder are orphaned (and `PruneCache`-reaped) rather than served or mixed at a boundary with new ones. Per-(asset,segment) in-flight registry (concurrent callers share one build); temp-file + atomic rename so a partial segment is never served; failed builds leave no segment so the next request retries. `PruneCache` bounds the cache under `DataDir/cache/tv-hls` (dirs touched recently protected by the grace window) and sweeps orphaned `.segNNNNN.ts.tmp` files left inside asset dirs by a hard-killed build (older than `buildTimeout`, comfortably past `segBuildTimeout` so in-flight builds are safe); its goroutine (`pruneTVCacheOnce`) is `recover()`-wrapped so a prune panic can't crash the process. **Audio warm-up at segment joins** (`buildSegment`/`audioWarmArgs`/`segmentMuxArgs`): re-encoding each segment's audio independently re-primes the AAC encoder, and that ~37ms of priming overwrites real audio at every 6s join (a faint per-6s click). So interior segments use a **two-pass warm-up** — pass 1 encodes audio from `segWarmupLead` (0.5s) *before* the boundary (the priming lands on the discarded lead-in, encoder warm by the boundary), pass 2 stream-copy-trims that lead-in (drops the priming, no re-encode) and muxes a fresh `-threads 3` video encode placed at the boundary; seg0 stays single-pass (its priming sits at negative PTS, decoder-trimmed). A ~30ms overlap still remains — independent segments can't tile exactly (AAC frames are 21.33ms, 6.0s = 281.25 frames, never integer) — but it's now near-identical **real** audio on both sides of the seam, not priming garbage (measured: warm join matches source level within ~0.2 dB vs the old ~2 dB priming dip). `segEncodeVersion` covers the warm-up bytes. **Audio filter — gap-fill + dialogue-forward downmix** (`audioFilterArgs`, shared by `SegmentArgs`/`audioWarmArgs`/`BurnInArgs`): every audio-encode path is fronted by **`aresample=async=1`**, which **silence-fills gaps in the source audio** and keeps the output PTS-locked so each segment emits *exactly* its window of audio. Without it, a source missing audio packets (a corrupt region, a bad rip) yields a segment whose audio is short of its declared duration — an MSE audio-buffer **hole** the browser resyncs by slowing (pitching **down**) the audio, which sticks until a reload rebuilds the pipeline (the "deep voices" bug on Doctor Who S02E02: a real 2s source audio gap at ~26min → seg had 189 AAC frames vs 282). Filling it with silence keeps A/V in sync (near-transparent on gap-free audio); `segEncodeVersion`→5 covers the added bytes. Then the **dialogue-forward downmix**: a source with a centre channel (`>=6` channels: 5.1/6.1/7.1) folds to stereo with `pan=stereo|FL=0.7*FC+0.5*FL|FR=0.7*FC+0.5*FR` instead of a naive `-ac 2` — centre (dialogue) weighted above the fronts, surrounds/LFE dropped — so dialogue isn't buried 3 dB under the music when a browser that can't decode Dolby (Chrome/Firefox: E-AC-3 unsupported) transcodes it to stereo. Clip-safe (coeffs <1) and layout-agnostic (names only FC/FL/FR, present in every `>=6`-channel layout; gate is `>=6` not `>2` because a `pan` naming FC would *error* on a centre-less 3-5ch layout). Always-on, deterministic per file → no HLS cache-key change; the channel count threads from the stored probe (`audioChannels`) through `EnsureSegment`/`streamTVBurnIn`. `<6`-channel sources keep `-ac 2`. **Bitmap-subtitle burn-in is the exception to segment-on-demand**: bitmap subs are stateful across the timeline (display sets persist across boundaries), which per-segment input seeks drop, so `BurnInArgs` runs one continuous `[0:v][0:s:N]overlay` transcode from the start, streamed as a progressive fragmented MP4 via `StreamFFmpeg` — played like remux, so it is not *freely* seekable. **Both progressive paths (remux + burn-in) are resume-capable**, though: the player passes `?start=<pos>` and `RemuxArgs`/`BurnInArgs` input-`-ss` there with `-avoid_negative_ts make_zero` (output timestamps rebased to zero); the player tracks `streamStartOffset` and reports `currentAbsTime` so the saved position stays on the real episode timeline (fixes resume restarting at 0 on remux/burn-in files). Remux (copy) lands within ±1 GOP of the request (bounded, non-drifting); burn-in (re-encode) is exact. Free fast-forward/rewind on these two paths is a future enhancement (pending.md Plan B). **Full-length scrubber on the progressive paths** (`StreamFFmpegPatchMoov` → `moovDurationPatcher`, `moovpatch.go`): `empty_moov` writes the `mvhd`/`tkhd`/`mdhd` durations as 0, so a plain `<video src>` can't show the real length — the scrubber only grows as it buffers (it "steps up" toward the true length in Firefox). The wrapper buffers the leading `moov` (tiny, written first), rewrites those duration fields in place to the **stream** duration (`total − start`, since a resumed stream is rebased to zero and carries only the tail), then streams the rest through untouched. Fixed-width overwrites only (no box resize → downstream offsets stay valid); patches only positively-identified boxes; passthrough when the duration is unknown or no `moov` appears. Verified in Firefox 152 + Chrome 149 (unpatched steps 10→60s; patched is a stable 60s from first metadata). This fixes the duration *display*; the stream is still not freely seekable (no range support) — unchanged.
- **Endpoints** (`internal/web`): `GET /tv/playback-session` returns the decision + source URL + track lists; `/stream/tv/` (direct), `/stream/tv-remux/`, `/stream/tv-burnin/` (progressive bitmap-sub burn-in), `/stream/tv-hls/` (manifest+segments), `/stream/tv-subtitles/` (WebVTT). HLS asset names are regex-whitelisted; all paths go through `pathguard`. **Default-audio resolution**: when the client doesn't pin a track (`aud==0`) and the file's disposition-default audio **isn't** stream index 0, the session resolves `aud` to the default's concrete 1-based ordinal (`defaultAudioOrdinal`) *before* `FromProbe`/`streamURL`, and bakes `?aud=N` into the emitted URL (HLS propagates it into the segment URIs via `VODPlaylist`). This keeps the decision (`pickStream` evaluates the default) and the served track (`audioMap(N)→0:a:(N-1)`) on the same stream — otherwise `Decide` could green-light remux on the default's codec while ffmpeg copies a different, incompatible index-0 track. A default that already is index 0 leaves `aud==0` (URL unchanged). **Movies reuse this whole layer** via thin clones (`handlers_stream_movie.go`): `/movie/playback-session` + `/stream/movie-{,remux,burnin,hls,subtitles}` + `/movie/playback-progress`, identical logic over `movie_files` + `movie_playback_progress`. The `internal/playback` + `internal/video` layers and the HLS cache are shared unchanged — the cache is keyed by **source path** (`hlsKey`), not row id, so TV and movie file-ids never collide.
- **External subtitle search** (`internal/opensubtitles` + `handlers_tv_subtitles.go`): when a file has no deliverable text subtitle track, the player offers an OpenSubtitles search (only if a key is configured). `OSClient` mirrors `match.FanartClient` (`New(key, userAgent)`→nil without a key, own 1 req/s limiter, **key-only** auth — `Api-Key` + a **configurable `User-Agent`**: the UA must name a consumer app *registered with OpenSubtitles* (`AppName vX.Y`) or requests are **403'd**, so it's resolved per-call by `effectiveOpenSubtitlesUserAgent` (app_settings `opensubtitles_user_agent` → `HESPERA_OPENSUBTITLES_USER_AGENT` env → default `Hespera v1.0`) and editable in Settings → API Keys): `Search(parentTMDBID, season, episode, lang)` / `Download(fileID)`. `GET /tv/subtitles/search?file=&lang=` derives the **TMDB series id + season + episode from `tv_series_identities` server-side** (precise — uses the stored ids, not a fuzzy title query) → JSON candidates; `POST /tv/subtitles/fetch` (form: `file`, `file_id`, `lang`) → `Download` → GET the link (**SSRF guard: host must be `*.opensubtitles.com` over https**, `isOpenSubtitlesHost`) → `ffmpeg -f webvtt` → cache at `DataDir/subtitles/{osFileID}.{lang}.vtt` (temp+rename) → URL; `GET /tv/subtitles/get?file_id=&lang=` serves the cached WebVTT (`nosniff`). Cache-first so a re-watch spends no quota (100 downloads/day). `lang` is validated/lowercased (`normalizeLang`, defaults `en`). Player dialog reuses the `playlist-modal` couch-overlay pattern. Caveat: non-UTF-8 SRTs may render garbled (no `-sub_charenc` detection yet).
- **Art** (`tvArt`, `/art/tv/`): serves `poster`/`backdrop`/`season` images from `tv_series_art`. Season cards are the single source of truth via a cascade — own `season_poster` → series poster → `302` to `/static/tv-poster-placeholder.webp` — so the template always emits the `<img>` (no per-card gate). Many shows lack per-season posters on TMDB (empty `poster_path`), so the series-poster fallback is the common path.
- **Client**: a **shared controller** `web/static/media_player.js` drives both `tv_player.html` and `movie_player.html` — loaded once from the layout shell (defer) and auto-booted on `turbo:load` (like couch.js/subtabs.js, so it survives Turbo body swaps without a body-script race); it derives its endpoints (`/tv/*` vs `/movie/*`, the progressive-URL regex) from the video's `data-media-kind`. It calls the session, then plays via `<video>` (direct/remux/native-HLS) or vendored hls.js (`web/static/hls.light.min.js`, Apache-2.0). Single-rendition by design — HLS is for seeking, not adaptive bitrate (a ladder is a future enhancement). **B2 custom transport**: native `<video controls>` is dropped (its scrubber can't address the progressive remux/burn-in streams — `video.seekable==[0,0]`) for a custom row — rewind 10s / play-pause / forward 10s / mute+volume / fullscreen + a **click/drag seek bar** (`media-seekbar` partial; a seek does a native `currentTime` set on the byte/segment-seekable paths, a single `?start=` reload on the progressive ones, fired only on drag-release so dragging never thrashes). Focusable for couch (Arrow ±10s); fill/time track the real episode timeline (`currentAbsTime`). **Clicking the video frame toggles play/pause** (`togglePlay`, a `click` listener on the `<video>` element itself — overlay controls and the floating skip button are separate elements that never reach it, and `.media-captions` is `pointer-events:none` so a click through a caption still toggles; when controls are auto-hidden the overlay is `pointer-events:none`, so the whole frame is live). **Controls overlay the video** (`media-overlay` inside `.tv-player-video-wrap`, seek bar + transport + track pickers) and **auto-hide** after ~2.5s of inactivity — revealed on pointer/keyboard activity, never hidden while paused / mid-drag / hovering the overlay / a control **keyboard-focused** (`:focus-visible`, so a lingering mouse-click focus doesn't pin it open — matching the couch focus-visible system). As the overlay hides, a **mouse-focused `<button>`** is `blur()`'d (the hidden overlay is `opacity:0`+`pointer-events:none`, which blocks the pointer but *not* keyboard activation, so a button the mouse last clicked would otherwise re-fire on Space/Enter while invisible) — scoped to `<button>` so an open `<select>` popup is never closed and the scrubber/volume arrow behavior is untouched. The **fullscreen button targets the wrap** (not the page) so the overlay stays over the video and the controls are reachable in fullscreen without exiting. The track-pick `<select>`s **`blur()` after a pick** so they don't keep `:focus-visible` inside the overlay and pin the controls open. **Subtitles are self-rendered, not native**: the sidecar `<track>` is kept `mode='hidden'` (browser parses cues but paints nothing), and `media_player.js` computes the active cue itself from a **media-clock time** against `captionTrack.cues` — the frame's exact `metadata.mediaTime` from `requestVideoFrameCallback` where supported (frame-accurate boundaries), else `video.currentTime` via an unconditional `timeupdate` floor (the sole driver on non-rVFC browsers; also covers paused-seek) — and paints it via `getCueAsHTML` (preserves `<i>/<b>` + multi-line) into a single `.media-captions` overlay div, diffed by `lastCueKey` and cleared on Off. It **never reads `activeCues`/`cuechange`**. This sidesteps two MSE/hls.js bugs at once: (1) native cue boxes never garbage-collected across track add/remove (toggling subs off/on/off/on stacked stale lines) or on switch-off — one div + replace-on-change makes stacking/stranding structurally impossible; (2) the browser's TextTrack cue scheduler drifting *ahead* of the media clock over a long session (subtitles creeping early) — anchoring to the media clock the audio/video ride bypasses it. The rVFC loop is gated on a caption track being attached (no per-frame wake when subs are off) and torn down on `turbo:before-cache`. **Toggling a text subtitle is a sidecar swap, not a reload** — `loadFromSession(…, subtitleOnly)` re-fetches the session for the new VTT and swaps the `<track>` via `attachSubtitle`, **without** rebuilding the hls pipeline; only a **burn-in** sub (which changes the video stream → a `?sub=` burn-in URL) reloads via `attachSource` (the `subBurnIn` ordinal set, from the server's `.text` flag, decides). This removes the per-toggle re-buffer and — with the recovery below — the black-screen-on-sub-toggle bug. **The hls.js instance carries an `Hls.Events.ERROR` handler** doing the documented recovery: a fatal **media** error → `recoverMediaError()` (time-guarded: recover, then `swapAudioCodec()`+recover, then give up with a `modeLabel` message — no infinite loop), a fatal **network** error → `startLoad()`; so a transient MSE append/parse failure on a seek (`CHUNK_DEMUXER_ERROR_APPEND_FAILED`) self-recovers instead of leaving the screen black until a manual reload (fixes the seek-while-paused black screen). **hls.js is preferred over native HLS on the transcode path** — the branch is `session.protocol==='hls' && Hls.isSupported()` (NOT gated on `!nativeHLS`): the whole transcode player (seeking, ERROR recovery, self-rendered subs) is built on hls.js/MSE, and **modern Chrome (149+) reports HLS as playable** (`canPlayType('application/vnd.apple.mpegurl')` → `'maybe'`), so trusting that signal silently routed the stream to Chrome's **native** HLS and bypassed the entire hls.js player — whose native seek/resume wedges (below). Native `<video src>` HLS is the fallback only for MSE-less browsers (iOS Safari, `!Hls.isSupported()`). **Resume/seek then anchors the fresh hls pipeline via `startPosition`** (`new Hls({…, startPosition: clientSeek>0 ? clientSeek : -1})`) so hls.js loads the **target segment first** rather than loading seg0 and then jumping `currentTime` — the seg0-then-jump appends fragments **out of DTS order** on a fresh pipeline, which Chrome MSE rejects as `not in DTS sequence` (`CHUNK_DEMUXER_ERROR_APPEND_FAILED`), wedging every resume-into-position on a transcoded (esp. HFR) file dead-on-arrival; loading the target first keeps appends in order, and the ERROR handler above is the backstop (`onReady`'s `currentTime` set stays — load-bearing for the native/progressive branch). The same row holds a **"Dialogue boost" toggle** (`#tvBoostBtn`, `icon-audio-lines`): a client-side Web Audio `DynamicsCompressorNode` + makeup `GainNode` taps the `<video>` element's decoded audio (`createMediaElementSource`, once per element) and compresses dynamic range to lift quiet dialogue under loud music/SFX. Works on **every** playback path (direct/remux/HLS) with no server transcode and full seeking preserved. **On by default** (`localStorage.tv_boost` is read as on unless explicitly `'0'`; the markup ships `is-on`/`aria-pressed=true` so there's no off→on flash); its **on** state is a **solid-accent fill** (`#tvBoostBtn.is-on` in `app.css` — `.btn-primary`'s tint was near-invisible over the overlay in light mode). Routing engages only when the `AudioContext` is `running`, so an autoplay-suspended context never silences playback (it passes through, then re-routes on `statechange`/`play`); the context is closed on `turbo:before-cache`. Evens *dynamics* only — the **server-side dialogue-forward downmix** (above) handles the complementary 5.1-centre-channel problem on the transcode/burn-in paths, before the audio reaches the browser as stereo.
- **Skip intro / recap / commercial** (`internal/web/skipsegments.go`): marker-based, no detection subsystem. `video.Probe` now captures **chapters** (`-show_chapters` → `ProbeResult.Chapters`, persisted in `stream_info_json` on scan); `skipSegmentsFor(probe, cleanPath)` builds `[]skipSegment{StartSec,EndSec,Kind}` from (1) chapter **titles** classified by regex (`intro|opening|title sequence`→intro, `recap|previously`→recap, `commercial|advert|ad break`→commercial; a plain "Chapter 1" never matches) and (2) a sibling **`<file>.edl`** sidecar (comskip/Kodi format — `START END [ACTION]`, actions 0/3 = commercial; the reliable commercial source since chapters rarely carry ads). Built **once per playback session** (off the hot path), emitted in the session JSON as `skip_segments` (TV + movie, pathguard-resolved sidecar). A **third source** is **audio-fingerprint intro detection** (`internal/introskip`, for files with no markers — the common case): `video.Fingerprint` extracts a Chromaprint fingerprint per episode via ffmpeg's built-in `chromaprint` muxer (**no new binary dep**; `video.ChromaprintAvailable` gates graceful degrade), and `introskip.DetectIntros` cross-matches a season's episodes (bounded brute-force shift + per-point Hamming run-length) to find the shared title theme at each episode's own offset — stored in `tv_skip_segments(file_id,kind,start_sec,end_sec,source)` and merged into the session by `dbTVSkipSegments`. On-demand via a **"Detect intros"** `data-live-job` button on `tv_series.html` (`POST /tv/series/detect-intros` → `intro_detect` job, per-season, ≥2 episodes). Verified on real content (Doctor Who S02: 8/8 episodes, ~33s theme, offsets 19s–398s — hence the 600s fingerprint window). Best-effort: episodes with no confident (≥15s) match get no intro, never a false skip. End-credits detection + auto-on-scan + cross-episode fingerprint caching are deferred (pending.md). Client (`media_player.js`): a floating **"Skip …"** button (`.media-skip-btn`) shows while `currentAbsTime` is inside a segment and seeks to its end (reusing the seekable/progressive seek); the `#skipAutoBtn` toggle (`localStorage skip_auto`, per-device, default off, revealed only when a file has segments) makes it **automatic** — auto-skipping each segment **once** (a `skippedAuto` guard so a deliberate manual rewind isn't fought). _Chapter/EDL markers are inert on the current library (no files carry them) but fingerprint detection works on it; verified via the fabricated chaptered mkv + `.edl` fixture and the real Doctor Who season._
- **Missing seasons/episodes**: `tvSeriesDetail`/`tvSeasonDetail` diff the cached TMDB metadata against present (matched) files and grey out the gaps — pure `missingSeasons`/`missingEpisodes` helpers, no new endpoints/queries/fetches. Specials (season 0) excluded. Reflects last-match-time cache (no live refetch).

### Cast / People

- **Data** (`people` + `credits` tables): `people` is a global TMDB-person cache (`tmdb_id` PK, name, `art_path`, bio, `bio_fetched_at`); `credits` links a person to a title — `(person_id, media_type['tv'|'movie'], media_id, character_name, billing_order)`, `media_type` discriminator so the future movie scanner reuses the set. Populated by `tmdb.FetchTVCast` (called inside `RunTVMatch` + `FetchShowMetadata`).
- **Backfill for already-matched shows**: `RunTVMatch` only processes *unmatched* rows, so existing series never get cast from match alone. `tvSeriesDetail` enqueues a background `tv_cast_fetch` job when a series has no `show:%d:cast` marker (the marker is written even for cast-less shows, so it doesn't re-enqueue every view). All metadata fetches run in background jobs — page handlers never block on TMDB (deduped via `Handler.metaFetch`, `enqueueMetaFetch`). **Movies mirror this**: `movieDetail` enqueues a `movie_cast_fetch` job gated by a `movie:%d:cast` marker (written in `FetchMovieCast`, in `movie_metadata_cache`), and `moviesMatchApprove` is **async** — it marks the rows matched then enqueues a `movie_metadata_fetch` job via **`enqueueMovieMetaFetch`** (the movie twin of `enqueueMetaFetch`, building a `NewMovieMatcher` so art lands in `thumbs/movies`, not the TV `thumbs/tv` the TV matcher would use). A per-movie **Unmatch** control (`POST /movie/unmatch`, button on the detail page) resets a mis-matched film to the review list + drops its `movie_art` — the movie analogue of `musicAlbumUnmatch`. A per-movie **manual art upload** (`POST /movie/art`, the "Artwork" panel on the detail page) overrides the TMDB poster/backdrop — bytes-sniffed (jpeg/png/webp, 15 MiB), temp+rename under `thumbs/movies` as `manual-movie-<tmdbid>-<type>`, upserting `movie_art` with a new **`manual=1`** flag. `downloadMovieArt` **gates on that flag** (early-returns when `manual=1`) so a (re)match never clobbers a manual upload, while TMDB art (`manual=0`/no row) stays refreshable — Unmatch→re-approve still fixes a wrong poster. `POST /movie/art/clear` reverts one slot (deletes the manual row + file, re-fetches TMDB art via `enqueueMovieMetaFetch`); `movieArt` serves the user bytes with `nosniff`. Unlike music's empty-only guard, movies need the explicit flag because auto-art is always non-empty *and* should refresh on re-match.
- **UI**: a cast strip on `tv_series.html` (carousel of `band-album-card--poster` cards: profile photo, actor name, character) below the overview, linking to `/person/{id}`. The actor page (`personDetail`, `person.html`) shows bio + image and the person's **full filmography split into two sections, "TV Shows" and "Films"** — every acting credit. Bio/image/filmography are fetched lazily via a background `person_fetch` job on first view (`FetchPersonBio` also calls `FetchPersonCombinedCredits` → `/person/{id}/combined_credits` — `cast` only, both media types — deduped/ordered newest-first by `topCredits`, **no cap**, cached as `people.filmography_json`). `buildFilmography` splits the blob by `media_type` and marks each credit **owned** against per-type ownership sets (TV: `credits`⋈`tv_series_identities` matched; film: `credits`⋈`movie_files` matched — keyed per type because a series id and a movie id can collide); owned titles link locally (`/tv/series/` or `/movie/`) with local art, **un-owned credits link out to their TMDB page** (`themoviedb.org/{tv,movie}/{id}`, new tab); the **bio carries a source link** — Wikipedia when its prose is attributed there (`splitWikipediaBio`), else the TMDB person page (`themoviedb.org/person/{id}`). **Lazy upgrade:** an actor cached before combined-credits — a non-empty `filmography_json` blob with no `media_type`, **or an empty `''` blob** (fetched before the column existed) — re-fetches once on next view (`filmographyNeedsUpgrade`). Loop-proof: a re-fetch result carries `media_type` or is `[]` (len 2 ≠ `''`), never `''` again; `[]`/`null` (fetched-empty) don't trigger.
- **Out-of-library posters are hotlinked** (`buildFilmography` builds `https://image.tmdb.org/t/p/w342{poster_path}` URLs for un-owned credits, `referrerpolicy="no-referrer"`) — a deliberate exception to the download-to-disk art rule, since these are external discovery thumbnails for titles the user doesn't own, not their media.
- **Images**: profile images downloaded to `thumbs/tv` (`person_%d_profile.jpg`), served by `/art/person/{id}` (`personArt`, pure serve-from-disk, `pathguard`+`nosniff`), GC'd by the thumbgc TV sweep (which now references `people.art_path`).

### Lyrics / Karaoke

- Synced lyrics via **LRCLIB** (free, no key). `POST /music/lyrics/fetch` (`handlers_music_lyrics.go`) is cache-first against the `lyrics_cache` table, else fetches (exact `get` then fuzzy `search` scored by `pickBestLrcLibCandidate`) and caches the result — **hits and misses both cached**, so a track is fetched at most once. Lazy per-track (no scan job). **Gated by the `lyrics_enabled` app-setting (default OFF, opt-in via Settings → API Keys, `effectiveLyricsEnabled`)** — the single source of truth for both the endpoint (returns immediately, no LRCLIB call) and the client (`musicPlayer` passes it as `.player-page[data-lyrics-enabled]`; `player.js` skips the fetch). **Per-song override:** the transport's Lyrics button (`#player-lyrics-btn`, `icon-mic-vocal`, `is-on` accent fill) flips lyrics for the current track only — an in-memory `lyricsOverride` map (track id → on/off) in player.js, session-scoped; an explicit "on" sends `force=1`, which the endpoint honors past the global gate (a deliberate user gesture, not an automatic fetch — still cache-first). The global setting stays the default for tracks without an override. `player.html` parses the synced LRC and advances current/next line on `timeupdate`. **No-lyrics layout:** when lyrics are disabled OR a track has no *synced* lyrics, `player.js` sets a `no-lyrics` class on `#player-main` — CSS (`.player-now.no-lyrics`) hides the `#player-karaoke` card and enlarges the cover so it expands into the freed space. **The card starts hidden and is revealed only once a track's synced lyrics are confirmed to exist** — `loadKaraokeForTrack` keeps `no-lyrics` on through the fetch and clears it *only* in the synced-result branch, so it never flashes an empty "Loading…" card for the many tracks with no synced lyrics (the trade is a brief cover shrink when lyrics do load). (No duration tiebreaker — match precision caveat tracked in pending.)

### Test Patterns

- Standard `testing` package with table-driven tests
- `openTestDB(t)` helper creates temp SQLite in `t.TempDir()`
- Direct conditionals with `t.Fatalf()`, no assertion libraries
- HTTP tests use `httptest.NewRequest` / `httptest.NewRecorder`
- **Browser-JS tests** (`test/js/`, `make test-js` / `npm test`): a **dev-only** harness — the Go binary embeds `web/` and never runs Node, so `package.json`/`node_modules` ship nothing and are `.gitignore`d; `jsdom` is the one devDependency (`npm install` once). `harness.js` loads a real `web/static/*.js` controller into a jsdom window (`vm.runInContext`) with the media engine stubbed (`HTMLMediaElement.play/load/canPlayType`, a mock `Hls`, `fetch`, `sendBeacon`) and exercises it **through its DOM effects** (dispatch `turbo:load`, click, change selects, fire `timeupdate`) — so the tests run the shipped code, not a copy. Covers `media_player.js` (config, selects, fmtTime, skip segments, HLS `startPosition`/error recovery, progressive `?start=`, volume/boost, progress) and `player.js` (autoload + local playback start + album-less cover guard). **Ceiling:** jsdom models the DOM, not the media engine — real MSE/DTS ordering, iframe reparent-reload, WebVTT cue parsing, and video decode stay in the Playwright smoke.
