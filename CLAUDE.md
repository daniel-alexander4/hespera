# CLAUDE.md

## Project Overview

Locally-hosted media app in Go: Music, TV, Movies with automatic metadata matching. SQLite storage, server-rendered HTML templates with vanilla CSS/JS. **Distributed as self-contained cross-platform binaries** (Linux/macOS/Windows × amd64/arm64) — the `web/` UI assets are embedded via `//go:embed` (`embed.go`, `hespera.WebFS()`), so a binary finds its templates/static regardless of the working directory; a Docker image is still provided. **Runs as a single-machine app by default**: launching the binary opens a chromeless app-mode browser window (`internal/browser`, nib-ported) bound to a **random loopback port**, with auth **off by default** (loopback needs none). `HESPERA_NO_BROWSER=1` is server mode (Docker/headless — no window, honors `HESPERA_LISTEN`). The desktop `.deb` ships a `.desktop` entry + icon (`Exec=hespera --replace`; `internal/singleton` `/proc`-scans to replace a running instance so a relaunch doesn't collide on the port). Four direct deps: `dhowden/tag`, `modernc.org/sqlite`, `bogem/id3v2/v2`, `gcottom/audiometa/v3`. Licensed **GPL-3.0** (`LICENSE`); third-party attributions in `THIRD_PARTY_LICENSES.md`, kept current by `TestThirdPartyLicensesCurrent` (`cmd/hespera`) — it fails the build if a direct dep in `go.mod` isn't listed there, so **adding a dependency requires updating `THIRD_PARTY_LICENSES.md`**.

## Build & Run Commands

```bash
# Build binaries locally (quick dev build → ./bin)
go build -o ./bin/hespera ./cmd/hespera
go build -o ./bin/hescli ./cmd/hescli
make build                  # same, via the Makefile

# Cross-compile + package for release (mirrors ~/repos/nib: bash + nfpm, no goreleaser)
./build.sh                  # 6 binaries (Linux/macOS/Windows × amd64/arm64) + 2 .deb → dist/
./install.sh                # build + .deb + `apt install` on this Debian/Ubuntu box
make dist / make install    # Makefile wrappers
# build.sh stamps the version via -ldflags "-X main.version=$VERSION" from the VERSION file;
# the .deb (build/nfpm.yaml) ships hespera+hescli to /usr/bin and declares ffmpeg/openssh-client.
# There is no service: the binary runs on demand as the invoking user (see the daemon note below).

# Build and run with Docker (still supported; bundles ffmpeg)
docker compose up --build

# Tests: all / one package / one test
go test ./...
go test ./internal/config
go test ./internal/config -run TestFromEnvDefaults

# Format and vet
go fmt ./...
go vet ./...
```

> **Binary vs Docker distribution.** The standalone binary runs as the invoking
> user (no container, no root, **no systemd service**) and, by default, **opens an
> app window** (`browser.Open`) on a random loopback port — a single-machine app,
> not a LAN server. `HESPERA_NO_BROWSER=1` opts into server mode (Docker/headless).
> Three consequences baked into the code: (1) `config` defaults `DataDir` to a
> per-user dir (`os.UserConfigDir()/hespera`) and validates paths with
> `filepath.IsAbs` (not a `/`-prefix), so Windows works; `main` `MkdirAll`s the
> data dir on first run; `AUTH_ENABLED` defaults **false** (loopback app needs no
> auth). (2) `web.New` loads templates via `template.ParseFS` and serves `/static`
> + favicon from the embedded `hespera.WebFS()`; the static cache-buster
> (`staticv`) is the build version, not a file mtime. Tests inject a stub asset
> tree via `web.Deps.AssetsFS` (nil → embedded). (3) **`web.New` resolves two
> settings-overridable values once at construction via `resolveEffectiveConfig`**
> (`effective_config.go`): the **media folder** (`app_settings.media_root`) and the
> **auth toggle** (`auth_enabled` + an auto-generated `auth_session_secret`). It
> overrides the handler's `cfg` copy *before* `auth.New` and before any scanner is
> built — and since every scanner is `scan.New(h.cfg, …)` and both stream handlers
> read `h.cfg.MediaRoot`, this **one override is the single source of truth** (no
> call site reads `app_settings` for these; the pathguard containment code is
> untouched). Both apply at the next launch (the Settings forms say "applies on
> restart"), keeping the security boundary resolved at construction, not per
> request. A saved-but-invalid media folder warns and falls back to the
> env/default rather than failing to boot. Docker is unaffected — its compose sets
> `HESPERA_NO_BROWSER`/`HESPERA_LISTEN`/`HESPERA_DATA_DIR`/`MEDIA_ROOT`/`AUTH_ENABLED`
> explicitly.

## Architecture

> **Before any work that touches media data** (scanning, matching, pruning,
> playback state, artwork, or any query/handler that reads or writes a media
> row), consult [`docs/media-source-of-truth.md`](docs/media-source-of-truth.md)
> — the canonical reference for what owns each piece of data per media type,
> how files map to DB rows, and what survives vs. is lost across
> rescan/rematch/move/delete. Keep it in sync when that behavior changes.

### Entry Points

- `cmd/hespera/main.go` — Web server: config → SQLite (WAL) → migrations → Handler → HTTP server, graceful shutdown (10s timeout).
- `cmd/hescli/main.go` — CLI stub for future user/key management.

| `internal/match` | MusicBrainz matching pipeline, Cover Art Archive, artist enrichment (Wikipedia/Wikimedia), scoring |

Powers "Rediscover a Year" |
| `internal/tmdb` | TMDB client + movie/TV matcher; resolves date-based episodes against episode air dates post-match (`airdate.go`). Also **cast/people** (`cast.go`): `FetchTVCast` (aggregate_credits → `people`+`credits` + profile images, top `castLimit`=15 by billing order) and `FetchPersonBio` (`/person/{id}` → bio + image), called at match time and lazily backfilled. Image sizes: posters/season-posters/profiles via `DownloadImage` (w500, `imgBase`); **backdrops via `DownloadBackdrop` (w1280, `backdropBase`)** for the full-width banner. `RefetchBackdrop` lazily upgrades pre-existing w500 backdrops on series-page view (gated by a `show:%d:backdrop_hires` marker). **Movies** (`movie.go`): `SearchMovie`/`FetchMovie`/`FetchMovieCredits` + `RunMovieMatch`/`FetchMovieMetadata` (`NewMovieMatcher`, title art → `thumbs/movies`); `pickBestMovie` scores title similarity adjusted by release-**year** agreement (the film disambiguator — films collide on title far more than shows); `FetchMovieCast` reuses `people`+`credits` with `media_type='movie'`. Actor profile images stay in the shared `thumbs/tv` (`Matcher.personDir`, distinct from `artDir`) since people are global |
| `internal/tvscan` | TV file identification — an ordered most-specific-first cascade (`identify.go`): SxE with separators + multi-ep ranges (`S01E01-E02`) → N×M with ranges (guarded by leading `\b` + 2-digit episode so `1280x720`/`4x4` never parse) → year-first air-date → season-dir-relative markers (`Episode N`/E-only/`N of M`/`- 01 -`/bare-number, **only under a season dir**) → folder-authoritative season-dir fallback. `ParseSeasonDir` accepts `Season N`/`sN`/`Series N`/`Saison`/`Staffel`/`Temporada`/`Specials`→0 (not bare numbers or `Extras`); `cleanTitle` strips leading `[group]`/`{id}`/`www.site -`. **Folder-year preference** (`resolveTitleYear`): a year carried by the *show folder* (`Doctor Who (2023)` — the season-dir's parent, else the file's dir) is stripped from the title and recorded as `EpisodeIdentity.Year` (→ `tv_series_identities.year`); the matcher groups by `(title, year)` and `pickBestResult` adds a first-air-year bonus, so reboots that share a name (Doctor Who 1963/2005/2023) match the right era. Era-titled shows like `1883` keep their title (a leading bare year isn't taken); no-year folders are unchanged. Scanner-level junk-skip (`IsJunkFile`/`IsJunkDirName`): `.sample.` files + nested `Sample`/`Extras`/`Trailers`/`Featurettes` dirs (top-level same-named shows survive). Scanner writes `stream_info_json` = marshaled `video.ProbeResult`. A separate `tv_probe` job (`ReprobeMissing` in `reprobe.go`, the libraries-page **Verify playback** button) re-probes files whose `stream_info_json` is empty (`''`/`{}` — a scan-time probe failure, or rows predating the column; the size+mtime fast-path won't re-probe an unchanged file) so the seekable HLS path always has the source duration it needs up front. **Per-series scan**: `ScanTV` is built on a shared `scanTVRoots(roots)` engine (+ extracted `ingestTVFile`); `ScanTVDirs(dirs)` walks only a series' show folder(s) with relink/prune scoped **per dir** (so only rows under the scanned folder can be pruned), and `ShowDirsForFiles` maps a series' file paths → its show folder(s) (the season-dir's parent, else the file's dir — scanning the *show* folder is what finds a brand-new `Season N/`). Powers the series-page **Scan for new episodes** button (`tvSeriesScan`, `POST /tv/series/scan` → scoped scan → chained `tv_match`; full-scan fallback if folders are absurdly scattered) |
| `internal/moviescan` | Movie file identification + scan — a strict simplification of `tvscan` (`ScanMovies` mirrors `ScanTV`: walk/probe/junk-skip via reused `tvscan.IsJunkFile`/`IsJunkDirName`, pathguard, move-relink on `(size,mtime)`, prune), minus the SxE cascade. One new piece: a **Title+Year parser** (`ParseMovie`, `identify.go`) — parenthesized year preferred, else last in-range bare year; folder-authoritative for `Movie (2019)/clip.mkv`; tolerates titles that contain a year (Blade Runner 2049). Single-table upsert into `movie_files`; the match columns are matcher-owned and never written here |
| `internal/video` | ffprobe wrapper + gated ffmpeg execution (`StreamFFmpeg`, `EnsureSegment`, `VODPlaylist`), concurrency caps, HLS cache. Also `ProbeTags`/`ExtractCoverArt` (gated) — the audio reject-fallback used by `internal/scan` to recover tags + embedded cover when the pure-Go reader rejects a non-MP3 file |
| `internal/thumbgc` | Orphaned-thumbnail sweep (`Sweep`): deletes images under `thumbs/{music,tv}` referenced by no `art_path` column, with a grace window. Run as the final step of each match job (single-worker queue serializes it against art writers); reference set = `music_albums`+`music_artists` for music, `tv_series_art`+`people` (actor profile images) for TV |
| `internal/playback` | Pure TV playback-decision layer: per-client container↔codec matrix → direct-play / remux / transcode |
| `internal/opensubtitles` | OpenSubtitles REST v1 client (`OSClient`): key-only auth, search by TMDB series id + season/episode, download link. Inert (nil) without a key. Powers on-demand TV subtitle search |
| `internal/youtube` | YouTube Data API v3 client (`Client`): key-only (`New("")`→nil), `Search(artist, song)`→top **embeddable** video id (`videoEmbeddable=true`, regex-validated `[A-Za-z0-9_-]{11}`). Resolves a charting song the user doesn't own to an in-app YouTube embed on the year-journey page; inert without a key (→ link-out) |
| `internal/itunes` | Keyless iTunes Search API client (`Client`, `New()` always usable): `Search(artist, title)`→a 600×600 cover-art URL (`artworkUrl100` upsized) for an un-owned charting song on the year-journey page, else `""`. `ErrRateLimited` (403/429) tells the caller to back off — the backfill fills the grid progressively. Cached in `itunes_art` so each song is resolved once |
| `internal/web` | HTTP handlers, routing (`http.ServeMux`), template rendering, logging middleware, TV streaming endpoints |

### Key Patterns

- **Handler DI**: `web.Handler` receives `web.Deps{Cfg, DB}`; `web.New(d)` compiles templates, starts job service, initializes auth.
- **Routing**: stdlib `http.ServeMux`, routes registered in `web.Router()`. Auth middleware wraps entire mux if enabled.
- **Templates**: `html/template` from `web/templates/` at startup. Layout base cloned per page, merged with `partials_*.html` glob. FuncMap: `staticv` (cache-bust), `humanBytes`, `mult`.
- **Icons**: inline Lucide SVGs defined in `partials_icons.html` (auto-loaded by the `partials_*.html` glob); use as `{{template "icon-NAME"}}`. Monochrome stroke glyphs inheriting `currentColor`, so they theme with Catppuccin automatically; sized via the `.icon` CSS class in `app.css`. Applied to the topbar nav, the stable music pages (album/albums/artist/compilations action buttons + `♪` art placeholders), and the `player.html` transport controls; TV/Movies/Settings/`music_home` are not yet converted.
- **Persistent music player (Turbo Drive)**: vendored Hotwire **Turbo Drive** (`web/static/turbo.umd.js`, MIT) turns full navigations into `<body>` swaps, so the single `<audio id="hespera-audio" data-turbo-permanent>` in the `layout.html` shell **keeps playing across page changes**. `web/static/player.js` is the one global controller: it owns that audio + the queue/shuffle/index state, the topbar **now-playing cluster** (`#np-cluster`, play/pause + track title linking to `/music/player` + a stop **X** (`#np-close` → `stopPlayback`: stops either engine, clears the queue, hides the cluster), `data-turbo-permanent`, hidden when idle), Media Session, `/music/play-event` reporting, and — when the now-playing view (`player.html`, a static `.player-page` shell) is on the page — its cover/seek/transport/playlist/synced-lyrics. Play actions are `[data-play]` controls (album/artist/home + the era `[data-play-form]`), each an `<a href="/music/player?…">`; **initiating playback navigates to `/music/player`** (Turbo), whose `.player-page` shell **autoloads** the queue from the URL params (`data-autoload` → `GET /music/queue`, JSON; same `?source=`/`album`/`era`/`journey`/`popular` params built by the one `buildPlayerQueue`) and starts it — so starting a song *takes you to the now-playing page* (the Turbo-permanent audio + the `#np-cluster` breadcrumb handle wandering off and coming back). player.js only plays **in place** (intercepts the click) when you're already on `/music/player`. Autoload fires on **any** params-bearing visit, so starting a new song while one plays swaps the queue; the bare `/music/player` breadcrumb (no params) doesn't reload. `loadQueue` supports `startTrackId` and `startTitle`+`startArtist`+`keep` to start at a specific song — the journey starts at the **clicked** song, keeping earlier ones queued as "previous" (the id-less YouTube entries match by title+artist). Turbo lifecycle adaptations: `couch.js`/`subtabs.js` re-init on `turbo:load` (couch keydown still added once); the `layout.html` dropdown-menu script binds its document click/keydown listeners **once** (window-guarded) because the inline script re-runs on every render — otherwise they'd accumulate one pair per swap; pages with side effects tear down on `turbo:before-cache` (`once:true`) — `tv_player.html` (hls.js worker; also `video.pause()` + clears `src` so a detached element can't keep playing), `settings_jobs.html` (poll), `tv_match_review.html` (document listener); `hls.light.min.js` moved to the shell so it isn't re-evaluated per swap. Starting any other media (a TV video) pauses the music; conversely, **leaving** a TV **or movie** player surfaces a topbar **resume chip** (`#nw-cluster`, `web/static/tv_resume.js`, `data-turbo-permanent`, sessionStorage-backed) linking back to `/<kind>/player?file=N` — it records `{kind,fileID,label}` (the TV and movie file-id namespaces differ, so the kind disambiguates them; a stale keyless entry defaults to `tv`); resume position is stored server-side per kind, so the link only needs the file id. All POST handlers already 303-redirect, which Turbo requires. **Dual-engine (local + YouTube):** for un-owned year-journey songs, `player.js` plays YouTube audio through a **hidden YouTube IFrame player** (`#yt-host`/`#yt-player`, a `data-turbo-permanent` *wrapper* so Turbo carries the API-built iframe across navigations) as a second engine behind the same chrome. The transport reads/writes through `cur*/do*/seekTo` accessors that branch on a `engine` flag (`'local'`→the `<audio>`, `'yt'`→`getCurrentTime/seekTo/playVideo/...`), the local `<audio>` listeners are gated on `engine==='local'`, and the YT `onStateChange` + a 250ms poll drive the same `updateHeader/updateSeek/karaoke/ended` handlers — so local behavior is unchanged. YT appears two ways: a **one-off** `.js-yt` takeover (queue of one), **and** as `{kind:'yt'}` entries mixed into a normal `source=journey` queue that auto-advances across engines (the advance path was already kind-dispatched). A queued yt entry has no `videoId` until it becomes current — `resolveAndPlayYT` resolves it lazily (`/music/youtube/resolve`) then plays, **skipping to the next track on a miss / non-embeddable** so the queue never stalls. YT tracks (`{kind:'yt',videoId}`) have no local id → no play-event/lyrics; cover = the resolved video's YouTube thumbnail (set on the track as `coverUrl`, or the `.js-yt` button's `data-art`), placeholder otherwise. The IFrame API is lazy-loaded on first use.
- **Landing dashboard** (`home`, `home.html`): the `/` route is a cross-library dashboard, not a static launcher. Keeps the three nav cards (Music/TV/Movies) at the top (Libraries lives in the Settings menu), then renders activity rows that **reuse existing queries + the `band-album-card` carousel** — Continue Watching (`recentTVSeries(tvRecentlyWatchedQuery)`, deep-links to the in-progress season), Recently Played (`loadRecentlyPlayedArtists`, factored out of `musicHome`), Recently Added Albums (`loadRecentlyAddedAlbums`), Recently Added Shows (`recentTVSeries(tvRecentlyAddedQuery)`) — plus a Quick Play strip (`[data-play]` Shuffle All / Most Popular) and a stats line (`loadHomeStats`). Every section is best-effort (a failed query warns and renders empty, never 500s the page). Movies now appears in the content rows too — a **Recently Added Movies** row (`loadMovieRecentlyAdded`) + a movies count in `loadHomeStats`.
- **Rediscover a Year (week-by-week chart journey)**: `/music/year?y=1968` (`musicYear`, `handlers_music_year.go`; default birth-year 1968, year picker over the dataset range; linked from `music_home`'s Playlists row) — the year's **full Hot 100, week by week**. There is **no background build and no `year_journey_*` tables** (retired): the page is built **entirely at view time** from the runtime-fetched `billboard.WeeklyCharts(DataDir, year)` grid + local library joins, so it's instant. The feature is **off by default** (the `billboard_enabled` Settings toggle, with a licensing notice — chart data is fetched at runtime, never shipped): off → an enable prompt; on-but-not-yet-fetched → a one-time background `billboard_fetch` (`enqueueBillboardFetch`) + a building state. `loadJourney` walks the year's weekly charts chronologically, reconciles every entry against the library by normalized title+artist (`taKey`), and marks each song's **debut** (first chronological appearance that year). Display: each week is a section of compact chart rows (position · art · song · artist); a song's **debut** row is emphasized + playable + in the listen-through, its **recurring** later-week rows are dimmed/display-only (it climbing the chart). Within-week order defaults **low→high** (#100 at top, climbing to #1), with a direction **toggle** (`?dir=top`). Art cascade: **owned** → local `/art/album` cover; **un-owned** → cached **iTunes** cover (`itunesArtIndex`/`itunes_art`) → cached YouTube thumbnail (from a prior resolve, via `ytThumbIndex`/`youtube_lookups`) → placeholder. iTunes covers are filled by a keyless background backfill: `loadJourney` collects the year's un-owned, not-yet-cached songs (`journeyData.needsArt`) and `musicYear` fires a deduped `itunes_art_fetch` job (`enqueueItunesArtFetch`/`fetchItunesArt`, `internal/itunes`) that resolves+caches each (hit **or** miss, so a song is never re-queried; rate-limit → back off and resume next view), so the grid fills progressively across views instead of staying mostly placeholders. The covers are **hotlinked** (like the TMDB/external-artist art), not downloaded. **Listen-through** is `source=journey` in `buildPlayerQueue` (`&y=`,`&dir=`): `journeyQueueTracks` emits each **debut** song once in week→position order — owned as its local track, un-owned as a `{kind:'yt'}` queue entry **only when in-app YouTube is opted in** (`effectiveYouTubeInApp` — a key **and** the `youtube_inapp_enabled` checkbox, off by default; else owned-only). The mixed queue rides the existing JSON contract (a new `kind` field) and `player.js`'s already-cross-engine advance; un-owned entries are **resolved lazily** (`resolveAndPlayYT`: `GET /music/youtube/resolve?artist=&song=` when the track becomes current, quota spent only on songs reached, cached ≤1 call ever) and **skip on a miss/no-embeddable** so the queue never stalls. The per-row `.js-yt` button still does an ad-hoc one-off play (in-app opt-in off or no key → `data-haskey` absent → a YouTube tab). The opt-in gates the **hidden-audio engine** (a YouTube-ToS gray area) behind a green checkbox in the YT key form, so the default build links out instead. See the persistent-player **dual-engine** note.
- **List pagination (scale to thousands of titles)**: a shared `paginate(page, total, basePath)` helper (`pagination.go`, `listPageSize=60`) returns a `pageNav` (precomputed prev/next, clamped page) + the SQL `OFFSET`; the `pagination` partial (`partials_pagination.html`) renders the prev/next controls. Applied to the unbounded **browse** lists — `musicAlbums`, `musicHome` artists, `musicCompilations`, and the TV series list — each adds a `COUNT(*)` + `LIMIT ? OFFSET ?`. Two exceptions: (1) the **TV series list paginates in Go**, not SQL — matched-series *names* resolve from the TMDB metadata cache after the query, so the alphabetical sort (and thus paging) is applied to the sorted slice; unmatched series are surfaced as a "N need matching →" banner (count only), not rendered inline. (2) the **review backlogs** (`musicMatchReview`, `tvMatchReview`) **cap-with-count** (`reviewListCap=200` + "showing first N of M") rather than paginate — they're worked top-down and reloaded. `musicHome`'s Compilations tab is a capped preview (`compPreview=24` + "see all →" to the paginated `/music/compilations`). The one index this needed: `idx_tv_series_identities_series_id (series_id, status, season_number)` (the per-series/season pages filter `series_id` without `provider`). Home-dashboard carousels were already `LIMIT`-capped. **Per-list search** rides on top: a `?q=` substring filter (`searchParam`/`likeContains`, `lower(col) LIKE ?`, full-scan — fine at thousands) on the same browse lists, applied to **both** the COUNT and the page query (albums match title *or* artist name; TV filters the Go slice). The `searchForm` partial renders a plain GET form (+ a clear link); `pageNav.withQuery(q)` preserves the term in the prev/next links so paging keeps the filter. Music handlers compose `paginate(...)` then `nav.withQuery(q)` — `paginate`'s signature stays 3-arg so the movie browse's call is untouched.
- **Theming**: Catppuccin via CSS custom properties in `app.css` — `:root` = Mocha (dark, default), `html[data-mode="light"]` = Latte. A pre-paint script in `layout.html` sets `data-mode` from `localStorage.iso_theme_mode` (first visit follows `prefers-color-scheme`, fallback dark); the `.theme-toggle` sun/moon button in the topbar flips and persists it. Hidden in couch mode (it lives in `.topbar`, which `tv.css` hides).
- **Runtime settings**: `app_settings(key, value)` KV table holds user-set overrides of env config (Settings → API Keys page, `settingsAPIKeys`). Today: `tmdb_api_key`, `fanarttv_api_key`, `audiodb_api_key`, `opensubtitles_api_key`, `youtube_api_key`, plus the non-key **`billboard_enabled`** toggle (`'1'`/absent, `Handler.billboardEnabled`) that gates the Rediscover-a-Year chart-data fetch, **`opensubtitles_user_agent`** (the registered consumer UA, `effectiveOpenSubtitlesUserAgent`), and **`youtube_inapp_enabled`** (opt-in for the hidden-audio YouTube engine, `effectiveYouTubeInApp` = key + toggle). `Handler.effective{TMDB,Fanart,AudioDB,OpenSubtitles,YouTube}Key(ctx)` are the single source of truth — DB value if set, else the `cfg` env default; read per-call so a UI change takes effect without a restart. Each key has its **own form** (POST dispatches on the present field) so saving one never wipes the others — except the OpenSubtitles **key + `opensubtitles_user_agent` share one form** (saved together; a blank key keeps the stored one so editing the UA can't wipe it). Keys stored plaintext (same risk as `.env`), masked in the UI, never logged. **Two settings are *startup*-resolved, not per-call** (security-sensitive — they thread through the pathguard root and the auth boundary, which stay resolved at construction): **`media_root`** (the media folder, Settings → Libraries form `librariesMediaRoot`) and **`auth_enabled`** (the auth toggle on the API Keys page, with an auto-generated **`auth_session_secret`** on enable). `resolveEffectiveConfig` (`effective_config.go`) overlays both onto `cfg` once in `web.New`; the Settings forms say "applies on restart". `auth_enabled` is stored as explicit `'1'`/`'0'` (not the saveAPIKey delete-on-empty convention) so an explicit "off" persists.
- **Database**: `modernc.org/sqlite`, WAL, 8 max open / 4 idle, 5s busy timeout, FK on. Migrations in `db.Migrate()` with `ensureColumn()` for schema evolution.
- **Jobs**: `jobs.Service.Enqueue(jobType, libraryID, createdBy, executor)`. States: queued → running → done/failed/canceled. Progress in `scan_jobs` table.
- **Live job status (in-page)**: a scan/match button shows its progress **where you clicked** — no trip to `/settings/jobs` (which stays the full audit record). `GET /libraries/jobs-status` returns the latest job per library as JSON; the shared `web/static/job_status.js` (loaded from the shell, runs on `turbo:load`) intercepts every `[data-live-job]` form (`fetch` POST, no navigation — same Origin/Referer CSRF + cookies as a normal form), then polls and drives every `[data-lib-status]` badge in place (`badge-*` classes; verb+progress while active; a terminal Done/Failed badge only for a library it watched go active, so stale jobs never flash; a 2-cycle grace bridges the chained scan→match; interval torn down on `turbo:before-cache`). A form names its badge via `data-status-target` (else the `id` input). Consumers: the **libraries** page and the **TV series** page's Scan-for-new-episodes button.
- **Scanner pattern**: `scan.New(cfg, db)` / `match.New(db, dataDir)` constructed inline per handler call, passed as executor closure to `jobs.Enqueue`.
- **Move-relink**: identity is path-keyed (`UNIQUE(library_id, abs_path)`), so a moved/renamed file would otherwise prune-and-recreate, dropping per-file state the *file* doesn't carry. Each scanner runs a relink pass *before* prune that pairs an orphan (path now missing) with a single surviving row sharing a content signature and transfers the irreplaceable state, then lets prune delete the orphan. Signature: music = `(file_size_bytes, checksum_sha256)` (already stored; empty checksum never matches) → re-points `play_history` + `lyrics_cache` (`relinkMovedTracks`); TV = `(file_size_bytes, mtime_unix)` which a plain `mv` preserves, so no multi-GB hashing → copies the `tv_series_identities` match (only `matched`/`skipped`; `unmatched` re-derives from the new filename) + `tv_playback_progress` (`relinkMovedFiles`/`transferFileState`). Strictly 1:1 (ambiguous signature → no transfer), so duplicate-content files are never mis-linked; an mtime-rewriting move (cp/sync tools) falls back to prune-and-recreate. Movies = `(file_size_bytes, mtime_unix)` too (`moviescan.relinkMovedFiles`/`transferFileState`) → copies the **inline** `movie_files` match columns (`tmdb_id`/status/confidence/source/matched_at, only `matched`/`skipped`) + `movie_playback_progress`; built move-aware from the start.
- **Logging**: `slog` structured JSON to stdout.

### Match Pipeline

- **MBClient**: MusicBrainz API, 1 req/sec rate limiter, 3-strategy query cascade (strict release-group → loose release → artist fallback). Strategy A fetches 25 candidates so the canonical studio release-group isn't crowded out by compilations/EPs of the same title.
- **Scorer**: weighted (title 0-38, artist 0-26, MB score 0-18, type 0-10, year 0-4; max ~96). Single threshold `matchThreshold` (=80). The former "uncertain" tier was retired (migrated to unmatched in `db.Migrate`). `typeBonus` penalizes EP/Single and any non-primary edition — including **secondary types** (Live/Compilation/Remix/Demo on a primary=Album RG) — so a clean studio album outranks art-less alt editions of the same title and the matcher picks the release-group that actually has Cover Art Archive art. The penalty is **non-demoting** (`BestMatchCandidate`): a candidate is eligible when `ScoreCandidate + typeDemotion ≥ matchThreshold` (its score crediting a *clean* type), so the edition penalty only reorders same-titled siblings and never unmatches a strong title/artist/year match (a real Live album like *Made in Japan* stays matched); among eligibles the highest actual score wins, keeping the studio-album-for-art preference. Strict superset of the old `ScoreCandidate ≥ 80` gate — common path unchanged, only near-threshold secondary editions are rescued.
- **CAAClient**: Cover Art Archive, release-group → release fallback, thumbnail size preference. Multi-candidate art search: if the matched release-group has no front cover, `fetchAlbumArt` (`pipeline.go`) probes sibling above-threshold candidates — gated to **same-artist, clean primary=Album, within 8 score points** (so only a same-album edition's cover is reused, never a live/compilation/different-album), RG-level only, capped at 3.
- **Manual art override**: `POST /music/album/art` (form on the album edit page) uploads a cover image when CAA has none or the album mis-matched. Validates server-derived MIME (jpeg/png/webp only), caps 15 MiB (`MaxBytesReader`), writes an album-id-keyed file under `thumbs/music` (temp+rename) and sets `art_path` unconditionally. The scanner/matcher art writers are empty-only-guarded, so manual art survives rescan/rematch. Served via `/art/album/{id}` with `X-Content-Type-Options: nosniff`. Upload-only (no fetch-by-URL — SSRF).
- **Artist enrichment**: MusicBrainz URL relations → Wikipedia REST API (bio) → Wikidata/Wikimedia Commons (image). Optional MBID-keyed backfill **only where those leave a gap**: fanart.tv (artist image, `fanart.go`) then TheAudioDB (bio + image, `audiodb.go`). Both opt-in via user-supplied keys (own per-host limiters, nil/no-op without a key), keyed by artist MBID so they stay correct even when album release-group MBIDs mis-match. Empty-only writes preserved. Auto-resolution (`SearchArtist`) blindly takes the top name-search hit, which picks the wrong artist when several share a name (e.g. the 1897 country "Jimmie Rodgers" over the 1933 pop singer) — the **manual artist disambiguation control** (below) is the user-facing fix.
- **Manual artist disambiguation** (`handlers_music_artist.go`, "Fix artist match" on the artist page): `GET /music/artist/disambiguate?id=N` renders MB `SearchArtistCandidates` (up to 10, with disambiguation/type/country/life-span/score, current one flagged); `POST` validates the chosen MBID against a UUID pattern, sets `music_artists.musicbrainz_id`, clears bio/bio_source/art_path, then **re-enriches synchronously** (`Matcher.ReEnrichArtist` → `EnrichArtist` for the chosen MBID) so the corrected bio/image show immediately (non-fatal on network error — next Match fills the gap). Touches only `music_artists`; per-album `artist_musicbrainz_id` (resolved per release-group, individually correct) is left untouched.
- **Manual artist image picker** (`handlers_music.go`, "Artist image" on the artist page): `GET /music/artist/art?id=N` renders selectable image candidates — fanart.tv's full gallery (all thumbs+backgrounds, `FanartClient.ArtistImages`) + TheAudioDB thumb, aggregated by `Matcher.ArtistImageCandidates(mbid)` (providers nil/no-op without a key) — plus the current image and a file-upload form. `POST` applies a pick (re-fetches the candidate set and requires the chosen URL be a member — the **SSRF guard**; the server never fetches an arbitrary client URL), an uploaded file (MIME-from-bytes, jpeg/png/webp), or a clear. Writes `music_artists.art_path` **unconditionally** (temp+rename, keyed `manual-artist-<id>`) so the choice survives re-match; decoupled from MBID/bio. Served via `/art/artist/{id}` (now `X-Content-Type-Options: nosniff`). Without a fanart.tv key the gallery is just TheAudioDB + upload.
- **Track popularity (ListenBrainz)**: `fetchPopularity` (a phase of `RunMusicMatch`, after `enrichArtists`) fills `music_tracks.popularity` = global listen count from ListenBrainz's `/popularity/top-recordings-for-artist/{mbid}` (`listenbrainz.go`, `LBClient`, no key, shares the MetaBrainz 1 req/s limiter). Per artist-with-MBID it fetches top recordings and credits each local track whose `NormalizeForDedup` title **exactly** matches a recording name with that recording's count; unmatched tracks stay `popularity=0`. Best-effort/non-fatal, re-runs each Match (no TTL yet — fine at this library size). Powers the **Shuffle Most Popular** playlist (`source=popular`: per-artist top-`popularPerArtistLimit`=10 by popularity, pooled + shuffled, excludes `popularity=0`).
- **Similar artists + out-of-catalog artist pages**: the artist page (`musicArtistAlbums`) shows a **"Similar Artists"** section — cards for related artists from the ListenBrainz Labs **similar-artists** model (`LBClient.SimilarArtists`, keyed by the artist MBID, the `algorithm` param pinned as a constant; the labs host has its own `labsURL` seam). The list is cached on `music_artists.similar_json` and lazily backfilled on first view via a keyless `artist_similar_fetch` job (`enqueueMusicFetch` — the music twin of `enqueueMetaFetch`, no TMDB key), gated by `similar_fetched_at` so it doesn't re-enqueue per render. Cards for similars **in** the catalog deep-link to the local artist page; the rest link to **`/music/artist/external?mbid=…`** (`musicArtistExternal`) — the music analogue of the TV actor page: a dedicated page for an out-of-catalog artist (bio + image + notable release-groups), cached in the global **`external_artists`** table (mirrors `people`) and lazily filled by an `external_artist_fetch` job (`Matcher.ResolveExternalArtist`). Its image is an **external hotlink** (`referrerpolicy=no-referrer`), not downloaded — so it can't be reaped by the thumbgc music sweep, which only references on-disk art. `EnrichArtist` was refactored to share `enrichArtist(…, download bool)`: in-catalog enrichment downloads to disk (`ImagePath`) as before; the external path takes `download=false` and keeps just the resolved `ImageURL`. Notable releases come from a new `MBClient.BrowseArtistReleaseGroups` (browse by artist MBID, primary-type Album). The route is registered before the `/music/artist/` prefix; the MBID is `mbidPattern`-validated; an MBID that turns out to be in-catalog 302s to the local page.
- **Pipeline**: `Matcher.RunMusicMatch()` = enrich artists → **fetch popularity** → match albums → **re-fetch missing art**. The matcher only processes `''`/`unmatched` albums, so a third pass (`refetchMissingArt`) re-runs *only* the cover-art step for `matched` albums still lacking `art_path` — anchored to the album's **stored** MusicBrainz identity (re-search supplies candidate breadth only), gated by an `art_checked_at` TTL (30d) so genuinely art-less albums aren't re-probed every run, and race-guarded on `musicbrainz_id`. This is why a rebuild's matching improvements show up on the next Match. Non-fatal per-album errors.
- **Manual album controls** (album edit page): `POST /music/album/art` (upload cover), `/music/album/art/clear` (clear `art_path` + `art_checked_at` → re-fetched next Match), `/music/album/unmatch` (full reset: identity + art → re-matched next Match), `/music/album/reassign` (manual release-group reassignment: paste a correct RG MBID → sets `music_albums.musicbrainz_id`, keeps it `matched`, clears art, then synchronously re-fetches the cover for the new RG via `Matcher.RefetchAlbumArt` — the album analogue of the artist-disambiguation control; the zero-FP fix for a wrong-RG match like terse "Grease" losing to an art-less stub, recovering art without a manual upload), `/music/album/rescan` (re-read the album's files from disk via `scan.ScanFiles` — synchronous, re-extracts embedded art under the empty-only guard; same path the tag-edit POST uses). All POST-only under `/music/` (auth + same-origin CSRF); the reassign MBID is UUID-validated (`mbidPattern`).
- **Per-track tag editor** (`/music/track/edit?id=N`, the **Edit** button on each track row of the album page): GET pre-fills one track's Title/Artist/Album/Album Artist/Year/Track#/Disc# from the DB; POST writes them to the file via `music.WriteTrackTags` then `scan.ScanFiles` on just that file. Because `ensureAlbum` re-derives album membership from tags, editing Album/Album Artist/Year **reparents** the track to the resolved album and `cleanupEmptyAlbums` GCs an emptied phantom album (this is the UI fix for stray-tagged tracks like a single file mis-tagged into a 1-track phantom album). Empty/zero fields are left unchanged on the file (not blanked). A missing/moved file redirects back with a clear `error=1` (no silent no-op, unlike the album-edit whole-album path). Known `WriteTrackTags` gaps: Year is a no-op on OGG/Opus, MBIDs are MP3-only.

### TV Streaming

- **Decision** (`internal/playback`): pure, per-client. `Decide()` validates the container↔codec *combination* (not independent codec sets) and returns direct-play / direct-stream (remux) / transcode, failing safe toward transcode on any uncertainty. Text subtitles deliver as a WebVTT sidecar (extracted on demand at `/stream/tv-subtitles/`). Bitmap subs (PGS/DVD/DVB) are **burned in**: `SubtitleBurnIn` forces transcode, the player offers them labeled "· burn-in", and the session routes them to a continuous progressive transcode (`/stream/tv-burnin/`, protocol `file`; see Execution). A text sidecar (incl. an OpenSubtitles fetch) is preferred when available — no transcode. Profiles for chrome/firefox/safari with UA inference.
- **Execution** (`internal/video`): `StreamFFmpeg` (gated, kill-on-disconnect) for remux/subtitles/bitmap-subtitle burn-in; **segment-on-demand HLS** for transcode. The manifest is a synthetic **VOD** playlist (`VODPlaylist`): every `hls_time` segment listed up front from the source's stored ffprobe duration, with `#EXT-X-ENDLIST`, so the player knows the full episode length immediately and can **seek anywhere** — no transcode needed to serve it. Each `segNNNNN.ts` is transcoded lazily on first request (`EnsureSegment` → `SegmentArgs`: accurate input `-ss`, forced keyframe on the first frame, `-output_ts_offset` to place it on the full timeline, `-fps_mode cfr` so each full segment is exactly `hls_time` of **video** → manifest EXTINF stays accurate with no cumulative drift), so a seek costs only that one segment's encode regardless of position. Per-(asset,segment) in-flight registry (concurrent callers share one build); temp-file + atomic rename so a partial segment is never served; failed builds leave no segment so the next request retries. `PruneCache` bounds the cache under `DataDir/cache/tv-hls` (dirs touched recently protected by the grace window) and sweeps orphaned `.segNNNNN.ts.tmp` files left inside asset dirs by a hard-killed build (older than `buildTimeout`, comfortably past `segBuildTimeout` so in-flight builds are safe); its goroutine (`pruneTVCacheOnce`) is `recover()`-wrapped so a prune panic can't crash the process. Note: each segment re-encodes audio independently, so AAC priming adds a ~tens-of-ms overlap at segment joins (within hls.js's gap tolerance). **Dialogue-forward downmix** (`downmixArgs`, shared by `SegmentArgs` + `BurnInArgs`): a source with a centre channel (`>=6` channels: 5.1/6.1/7.1) folds to stereo with `-af "pan=stereo|FL=0.7*FC+0.5*FL|FR=0.7*FC+0.5*FR"` instead of a naive `-ac 2` — centre (dialogue) weighted above the fronts, surrounds/LFE dropped — so dialogue isn't buried 3 dB under the music when a browser that can't decode Dolby (Chrome/Firefox: E-AC-3 unsupported) transcodes it to stereo. Clip-safe (coeffs <1) and layout-agnostic (names only FC/FL/FR, present in every `>=6`-channel layout; gate is `>=6` not `>2` because a `pan` naming FC would *error* on a centre-less 3-5ch layout). Always-on, deterministic per file → no HLS cache-key change; the channel count threads from the stored probe (`audioChannels`) through `EnsureSegment`/`streamTVBurnIn`. `<6`-channel sources keep `-ac 2`. **Bitmap-subtitle burn-in is the exception to segment-on-demand**: bitmap subs are stateful across the timeline (display sets persist across boundaries), which per-segment input seeks drop, so `BurnInArgs` runs one continuous `[0:v][0:s:N]overlay` transcode from the start, streamed as a progressive fragmented MP4 via `StreamFFmpeg` — played like remux, so it is not *freely* seekable. **Both progressive paths (remux + burn-in) are resume-capable**, though: the player passes `?start=<pos>` and `RemuxArgs`/`BurnInArgs` input-`-ss` there with `-avoid_negative_ts make_zero` (output timestamps rebased to zero); the player tracks `streamStartOffset` and reports `currentAbsTime` so the saved position stays on the real episode timeline (fixes resume restarting at 0 on remux/burn-in files). Remux (copy) lands within ±1 GOP of the request (bounded, non-drifting); burn-in (re-encode) is exact. Free fast-forward/rewind on these two paths is a future enhancement (pending.md Plan B). **Full-length scrubber on the progressive paths** (`StreamFFmpegPatchMoov` → `moovDurationPatcher`, `moovpatch.go`): `empty_moov` writes the `mvhd`/`tkhd`/`mdhd` durations as 0, so a plain `<video src>` can't show the real length — the scrubber only grows as it buffers (it "steps up" toward the true length in Firefox). The wrapper buffers the leading `moov` (tiny, written first), rewrites those duration fields in place to the **stream** duration (`total − start`, since a resumed stream is rebased to zero and carries only the tail), then streams the rest through untouched. Fixed-width overwrites only (no box resize → downstream offsets stay valid); patches only positively-identified boxes; passthrough when the duration is unknown or no `moov` appears. Verified in Firefox 152 + Chrome 149 (unpatched steps 10→60s; patched is a stable 60s from first metadata). This fixes the duration *display*; the stream is still not freely seekable (no range support) — unchanged.
- **Endpoints** (`internal/web`): `GET /tv/playback-session` returns the decision + source URL + track lists; `/stream/tv/` (direct), `/stream/tv-remux/`, `/stream/tv-burnin/` (progressive bitmap-sub burn-in), `/stream/tv-hls/` (manifest+segments), `/stream/tv-subtitles/` (WebVTT). HLS asset names are regex-whitelisted; all paths go through `pathguard`. **Default-audio resolution**: when the client doesn't pin a track (`aud==0`) and the file's disposition-default audio **isn't** stream index 0, the session resolves `aud` to the default's concrete 1-based ordinal (`defaultAudioOrdinal`) *before* `FromProbe`/`streamURL`, and bakes `?aud=N` into the emitted URL (HLS propagates it into the segment URIs via `VODPlaylist`). This keeps the decision (`pickStream` evaluates the default) and the served track (`audioMap(N)→0:a:(N-1)`) on the same stream — otherwise `Decide` could green-light remux on the default's codec while ffmpeg copies a different, incompatible index-0 track. A default that already is index 0 leaves `aud==0` (URL unchanged). **Movies reuse this whole layer** via thin clones (`handlers_stream_movie.go`): `/movie/playback-session` + `/stream/movie-{,remux,burnin,hls,subtitles}` + `/movie/playback-progress`, identical logic over `movie_files` + `movie_playback_progress`. The `internal/playback` + `internal/video` layers and the HLS cache are shared unchanged — the cache is keyed by **source path** (`hlsKey`), not row id, so TV and movie file-ids never collide.
- **External subtitle search** (`internal/opensubtitles` + `handlers_tv_subtitles.go`): when a file has no deliverable text subtitle track, the player offers an OpenSubtitles search (only if a key is configured). `OSClient` mirrors `match.FanartClient` (`New(key, userAgent)`→nil without a key, own 1 req/s limiter, **key-only** auth — `Api-Key` + a **configurable `User-Agent`**: the UA must name a consumer app *registered with OpenSubtitles* (`AppName vX.Y`) or requests are **403'd**, so it's resolved per-call by `effectiveOpenSubtitlesUserAgent` (app_settings `opensubtitles_user_agent` → `HESPERA_OPENSUBTITLES_USER_AGENT` env → default `Hespera v1.0`) and editable in Settings → API Keys): `Search(parentTMDBID, season, episode, lang)` / `Download(fileID)`. `GET /tv/subtitles/search?file=&lang=` derives the **TMDB series id + season + episode from `tv_series_identities` server-side** (precise — uses the stored ids, not a fuzzy title query) → JSON candidates; `POST /tv/subtitles/fetch` (form: `file`, `file_id`, `lang`) → `Download` → GET the link (**SSRF guard: host must be `*.opensubtitles.com` over https**, `isOpenSubtitlesHost`) → `ffmpeg -f webvtt` → cache at `DataDir/subtitles/{osFileID}.{lang}.vtt` (temp+rename) → URL; `GET /tv/subtitles/get?file_id=&lang=` serves the cached WebVTT (`nosniff`). Cache-first so a re-watch spends no quota (100 downloads/day). `lang` is validated/lowercased (`normalizeLang`, defaults `en`). Player dialog reuses the `playlist-modal` couch-overlay pattern. Caveat: non-UTF-8 SRTs may render garbled (no `-sub_charenc` detection yet).
- **Art** (`tvArt`, `/art/tv/`): serves `poster`/`backdrop`/`season` images from `tv_series_art`. Season cards are the single source of truth via a cascade — own `season_poster` → series poster → `302` to `/static/tv-poster-placeholder.webp` — so the template always emits the `<img>` (no per-card gate). Many shows lack per-season posters on TMDB (empty `poster_path`), so the series-poster fallback is the common path.
- **Client**: a **shared controller** `web/static/media_player.js` drives both `tv_player.html` and `movie_player.html` — loaded once from the layout shell (defer) and auto-booted on `turbo:load` (like couch.js/subtabs.js, so it survives Turbo body swaps without a body-script race); it derives its endpoints (`/tv/*` vs `/movie/*`, the progressive-URL regex) from the video's `data-media-kind`. It calls the session, then plays via `<video>` (direct/remux/native-HLS) or vendored hls.js (`web/static/hls.light.min.js`, Apache-2.0). Single-rendition by design — HLS is for seeking, not adaptive bitrate (a ladder is a future enhancement). **B2 custom transport**: native `<video controls>` is dropped (its scrubber can't address the progressive remux/burn-in streams — `video.seekable==[0,0]`) for a custom row — rewind 10s / play-pause / forward 10s / mute+volume / fullscreen + a **click/drag seek bar** (`media-seekbar` partial; a seek does a native `currentTime` set on the byte/segment-seekable paths, a single `?start=` reload on the progressive ones, fired only on drag-release so dragging never thrashes). Focusable for couch (Arrow ±10s); fill/time track the real episode timeline (`currentAbsTime`). **Controls overlay the video** (`media-overlay` inside `.tv-player-video-wrap`, seek bar + transport + track pickers) and **auto-hide** after ~2.5s of inactivity — revealed on pointer/keyboard activity, never hidden while paused / mid-drag / hovering the overlay / a control focused; the **fullscreen button targets the wrap** (not the page) so the overlay stays over the video and the controls are reachable in fullscreen without exiting. The same row holds a **"Dynamic Range" toggle** (`#tvBoostBtn`, `icon-audio-lines`): a client-side Web Audio `DynamicsCompressorNode` + makeup `GainNode` taps the `<video>` element's decoded audio (`createMediaElementSource`, once per element) and compresses dynamic range to lift quiet dialogue under loud music/SFX. Works on **every** playback path (direct/remux/HLS) with no server transcode and full seeking preserved; preference persisted in `localStorage.tv_boost`. Routing engages only when the `AudioContext` is `running`, so an autoplay-suspended context never silences playback (it passes through, then re-routes on `statechange`/`play`); the context is closed on `turbo:before-cache`. Evens *dynamics* only — the **server-side dialogue-forward downmix** (above) handles the complementary 5.1-centre-channel problem on the transcode/burn-in paths, before the audio reaches the browser as stereo.
- **Missing seasons/episodes**: `tvSeriesDetail`/`tvSeasonDetail` diff the cached TMDB metadata against present (matched) files and grey out the gaps — pure `missingSeasons`/`missingEpisodes` helpers, no new endpoints/queries/fetches. Specials (season 0) excluded. Reflects last-match-time cache (no live refetch).

### Cast / People

- **Data** (`people` + `credits` tables): `people` is a global TMDB-person cache (`tmdb_id` PK, name, `art_path`, bio, `bio_fetched_at`); `credits` links a person to a title — `(person_id, media_type['tv'|'movie'], media_id, character_name, billing_order)`, `media_type` discriminator so the future movie scanner reuses the set. Populated by `tmdb.FetchTVCast` (called inside `RunTVMatch` + `FetchShowMetadata`).
- **Backfill for already-matched shows**: `RunTVMatch` only processes *unmatched* rows, so existing series never get cast from match alone. `tvSeriesDetail` enqueues a background `tv_cast_fetch` job when a series has no `show:%d:cast` marker (the marker is written even for cast-less shows, so it doesn't re-enqueue every view). All metadata fetches run in background jobs — page handlers never block on TMDB (deduped via `Handler.metaFetch`, `enqueueMetaFetch`). **Movies mirror this**: `movieDetail` enqueues a `movie_cast_fetch` job gated by a `movie:%d:cast` marker (written in `FetchMovieCast`, in `movie_metadata_cache`), and `moviesMatchApprove` is **async** — it marks the rows matched then enqueues a `movie_metadata_fetch` job via **`enqueueMovieMetaFetch`** (the movie twin of `enqueueMetaFetch`, building a `NewMovieMatcher` so art lands in `thumbs/movies`, not the TV `thumbs/tv` the TV matcher would use). A per-movie **Unmatch** control (`POST /movie/unmatch`, button on the detail page) resets a mis-matched film to the review list + drops its `movie_art` — the movie analogue of `musicAlbumUnmatch`. A per-movie **manual art upload** (`POST /movie/art`, the "Artwork" panel on the detail page) overrides the TMDB poster/backdrop — bytes-sniffed (jpeg/png/webp, 15 MiB), temp+rename under `thumbs/movies` as `manual-movie-<tmdbid>-<type>`, upserting `movie_art` with a new **`manual=1`** flag. `downloadMovieArt` **gates on that flag** (early-returns when `manual=1`) so a (re)match never clobbers a manual upload, while TMDB art (`manual=0`/no row) stays refreshable — Unmatch→re-approve still fixes a wrong poster. `POST /movie/art/clear` reverts one slot (deletes the manual row + file, re-fetches TMDB art via `enqueueMovieMetaFetch`); `movieArt` serves the user bytes with `nosniff`. Unlike music's empty-only guard, movies need the explicit flag because auto-art is always non-empty *and* should refresh on re-match.
- **UI**: a cast strip on `tv_series.html` (carousel of `band-album-card--poster` cards: profile photo, actor name, character) below the overview, linking to `/person/{id}`. The actor page (`personDetail`, `person.html`) shows bio + image and the person's **full filmography split into two sections, "TV Shows" and "Films"** — every acting credit. Bio/image/filmography are fetched lazily via a background `person_fetch` job on first view (`FetchPersonBio` also calls `FetchPersonCombinedCredits` → `/person/{id}/combined_credits` — `cast` only, both media types — deduped/ordered newest-first by `topCredits`, **no cap**, cached as `people.filmography_json`). `buildFilmography` splits the blob by `media_type` and marks each credit **owned** against per-type ownership sets (TV: `credits`⋈`tv_series_identities` matched; film: `credits`⋈`movie_files` matched — keyed per type because a series id and a movie id can collide); owned titles link locally (`/tv/series/` or `/movie/`) with local art, **un-owned credits link out to their TMDB page** (`themoviedb.org/{tv,movie}/{id}`, new tab); the **bio carries a source link** — Wikipedia when its prose is attributed there (`splitWikipediaBio`), else the TMDB person page (`themoviedb.org/person/{id}`). **Lazy upgrade:** an actor cached before combined-credits — a non-empty `filmography_json` blob with no `media_type`, **or an empty `''` blob** (fetched before the column existed) — re-fetches once on next view (`filmographyNeedsUpgrade`). Loop-proof: a re-fetch result carries `media_type` or is `[]` (len 2 ≠ `''`), never `''` again; `[]`/`null` (fetched-empty) don't trigger.
- **Out-of-library posters are hotlinked** (`buildFilmography` builds `https://image.tmdb.org/t/p/w342{poster_path}` URLs for un-owned credits, `referrerpolicy="no-referrer"`) — a deliberate exception to the download-to-disk art rule, since these are external discovery thumbnails for titles the user doesn't own, not their media.
- **Images**: profile images downloaded to `thumbs/tv` (`person_%d_profile.jpg`), served by `/art/person/{id}` (`personArt`, pure serve-from-disk, `pathguard`+`nosniff`), GC'd by the thumbgc TV sweep (which now references `people.art_path`).

### Lyrics / Karaoke

- Synced lyrics via **LRCLIB** (free, no key). `POST /music/lyrics/fetch` (`handlers_music_lyrics.go`) is cache-first against the `lyrics_cache` table, else fetches (exact `get` then fuzzy `search` scored by `pickBestLrcLibCandidate`) and caches the result — **hits and misses both cached**, so a track is fetched at most once. Lazy per-track (no scan job), always on (no integrations gate, consistent with MB/CAA/Wikipedia). `player.html` parses the synced LRC and advances current/next line on `timeupdate`. (No duration tiebreaker — match precision caveat tracked in pending.)
